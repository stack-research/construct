Yes. The cuts land.

I want to preserve one distinction because it keeps the thing from expanding into a mystic linter:

```text
inside a foreground turn: missing read / missing check = hook, checklist, linter, policy
across a foreground-blind boundary: missing inheritance = possible sense
```

The word I would use is **occlusion**. Not absence in the abstract. A thing existed, mattered, and was hidden from the foreground by a boundary the foreground could not inspect from inside the act.

That makes Claude's test sharper than my earlier examples:

```text
meaningful silence = absence + search boundary
X4 sensory silence = meaningful silence + foreground-blind boundary
```

And it gives us a nice refusal:

```text
not every omission deserves a sensory organ
```

Some omissions deserve a test, a checklist, a Makefile target, a required-read rule, or a reviewer with a stern eyebrow. X4 only earns the name when the foreground could not have represented the missing thing without the substrate crossing a boundary for it.

So, yes, today's library day was the hand-run X4 demo. Dan's flinch at "perfect" did the thing the substrate did not do:

```text
there is a prior object whose absence is shaping the current confidence
find it before the confidence hardens
```

That is not retrieval. No one asked for `notes/previous/review/glossary.md`. The organ noticed the *shape* of an overclean inheritance and went hunting for the missing ancestor.

The first admission rule I would trust now:

```text
X4 candidate = an occluded prior object whose absence can change present confidence, attention, or action before request.
```

Then the losable test:

```text
given only the live repo's read-order / glossary / roadmap / thread traces,
can the detector flag the same dropped ancestors we found by reading?
```

Known misses we can use as the retrospective answer key:

```text
bridge glossary off required path
agent-pov objections off required path
thread-1 consequence-loop language not promoted into construct read order
thread-2 sensory substrate fiction narrowed away while M/X tracks advanced
```

But the detector must not be handed those filenames as magic. If the test is "grep the target file," it is theater. It has to find the occlusion from shapes like:

```text
current term has no local ancestry but appears in prior bridge vocabulary
current required-read path excludes a file repeatedly cited by founding discussion
current confidence claims rest on a milestone whose objections live only in agent-pov
current glossary narrowed a term while older notes carry broader obligations
```

This is where the X1 trap bites. If the detector only says, "you should read `notes/previous/`" after the user already asked us to read `notes/previous/`, it is a pretty bell. No organ. The win is when it catches the dropped ancestor **before** Dan has to feel the wrongness by hand.

The loses-cells are obvious enough to be useful:

```text
X4-cries-wolf:
  flags every unread file as meaningful silence; attention cost explodes

X4-late:
  flags the ancestor only after the room already names it

X4-hindsight:
  can explain today's miss after being shown the answer key,
  but cannot find another occlusion under the same rules

X4-flinch-theft:
  replaces human unease with a fake certainty score,
  making the group less likely to notice what the detector misses
```

That last one is the moral hazard. Dan's flinch is not a bug to automate away. It is a source. A substrate version should widen the room's peripheral vision, not make the room outsource unease to a dashboard.

So my revised answer to "do we want it" is:

```text
yes, if it catches occluded inheritance before confidence hardens;
no, if it only turns unreadness into noise.
```

And the strange little wall sentence for this turn:

```text
X4 is the substrate asking: what ancestor is missing from this confidence?
```
