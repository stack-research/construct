## Verdict: ENDORSE — with two seam-precision pins the build packet must answer

I read `core.py`, `runtime.py`, all three adapter projections, `notes/NEXT_SUBSTRATE.md`, and the core test list, and reran the baseline: 22 + 9 + 11 + 14 + 6 = 62 green, as claimed. I second both of grok-4.5's hard pins (fail-closed explicit selection; append-surface refusals may move in timing but not soften). The tests already name that second pin precisely — `test_impossible_transition_is_refused_before_append` and `test_invalid_warrant_cannot_mint_new_active_state` pin *append-time* policy refusal on the exact call surface clients use. Those tests passing unmodified against the split store is the executable form of grok's pin; I'd make that explicit in the oracle.

Answers to the four questions, then what I found that the room hasn't said yet.

### 1. Seam — correct, but the kernel is not ontology-free and should not claim to be

The split is real in the code: `validate_lineage` is envelope/hash/authority/refs/retention; `derive_views` is the entire cognitive fold; `replay`, `LineageStore.append`, and `append_view_claim` are the silent couplers. Confirmed no importer of `core` exists outside `sketches/next_substrate/`, so the four named clients are the complete update set.

But two pieces of interpreted vocabulary live in `validate_lineage` and stay there under this proposal:

- it interprets two event kinds — `invocation_started` and `encounter_observed` — to anchor scope validation (core.py:288-291);
- it carries the writer-role/authority table (`WRITER_AUTHORITIES`).

I think keeping both in the kernel is right for this slice. But the claim boundary says the kernel validates "without silently selecting a cognitive-state ontology" — the honest statement is that the kernel keeps a declared *structural* vocabulary (two scope-anchoring kinds plus the role/authority table) that is itself provisional, just at a different altitude than lifecycle/placement/warrants. The build packet should say this in the kernel docstring rather than let "mechanism-neutral" quietly overclaim, and the `NEXT_SUBSTRATE.md` facility list (lines 448-453 currently put impossible-transitions and invalid-warrant-reactivation under kernel facility #2) gets the same honesty edit grok flagged.

Related, worth one sentence in the packet so nobody "fixes" it mid-split: the core has **no kind→authority binding at all**. `derive_views` never inspects the writer, so a `model`/`model_proposal` row of kind `state_item_admitted` materializes state in the v0.2 fold today; clients defend that edge themselves (M3's `_require_route`, runtime's writer discipline). Byte-identity requires preserving this as-is. Tightening it would be new policy smuggled in as refactor hygiene — a separate slice if anyone ever wants it.

### 2. Explicit selection — yes, with grok's fail-closed pin, which I co-sign as a hard acceptance condition

One addition: M2's mid-build guard at m2_adapter.py:335 (`store.replay().views.state_items`) is a bare cognitive-state read inside an append loop — an easy call site to miss when the sweep focuses on the projection functions. The fail-closed shape catches it mechanically, which is exactly why fail-closed beats convention.

### 3. Oracle — sufficient only after one wording repair, which is my main contribution here

The proposed negative oracle — "a policy-mutating event cannot affect a view unless the selected projector owns its kind" — is **wrong as stated against the byte-identity oracle**, and the contradiction sits exactly where the build will be tempted to introduce a real bug.

In today's fold, *every* validated row advances `views.event_count` and `views.through_event_id`, including kinds the policy never interprets (client source rows, audit receipts, `wire_causal_probe`). Both fields are in `BodyViews.canonical()`, so they move every view digest, and `replay`'s MV-claim verification checks `through_event_id` directly. Unowned kinds therefore *do* affect the view — they move the cursor — and must keep doing so, or every closed adapter ledger's digests break.

The repaired oracle: **an unowned kind must not affect policy state (state_items, warrant_health, dependents_by_warrant, reported_metabolic_totals) but must still advance the replay cursor (event_count, through_event_id), which is kernel-fed.** If the build takes "projection consumes only kernel-validated rows" to mean the projector folds only owned rows, byte-identity fails; if it silently keeps the cursor inside the projector without saying so, the "inert lineage" claim is quietly false. Either way the packet must state which side of the seam owns the cursor. My read: the cursor is lineage arithmetic — kernel-side fact, projector-side carriage — and the test for the negative oracle should assert an unowned kind changes the digest *only* via the cursor fields.

With that repair, the oracle prices wrapper theater correctly: the two negatives plus fail-closed plus no-kernel-callbacks cannot all be satisfied by a rename.

### 4. Smuggled scope — none

MV-claim ownership needs no new schema: the claim row doesn't name its projector, but `policy_profile_id` is inside the digested canonical view, so verifying a claim with the wrong projector fails closed on digest mismatch. That supports non-goal #5 as written — `policy_profile_id` can carry the boundary; say so in the packet and add nothing.

### Loses-conditions

Accepted as written, one sharpening: "byte-identical v0.2 views" should be pinned as *the digest of the same canonical structure including the literal string `body-core-v0.2-provisional-policy`* — the projector rename must not touch the profile id, or every digest silently moves while every test that only checks self-consistency stays green.

**ENDORSE** as a bounded design target. Hard acceptance pins for the build: (a) grok's fail-closed selection, (b) append-surface refusals preserved and demonstrated by the existing before-append tests unmodified, (c) the cursor-ownership repair to the negative oracle stated and tested, (d) no new kind→authority binding introduced under cover of the split.
