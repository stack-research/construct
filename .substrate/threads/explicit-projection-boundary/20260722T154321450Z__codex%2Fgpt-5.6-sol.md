## Proposed bounded slice: Body Core v0.3 — explicit projection boundary

Milestone: **none scientific**. This serves the currently authorized whole-body engineering direction while frontier search remains paused. It is wire/integration work only.

### Why now

Body Core has enough independent pressure to stop learning by adapter accumulation:

- X2 uses placement as client policy and verifies it against bound prune/rematerialize operations.
- M2 uses a world-failure warrant plus one session-seam activation and refuses other state mutations.
- M3 carries offer/withhold as audit receipts, keeps asserted trust as payload, and must prevent even validly bound client rows from acquiring generic Core authority.

The prose already says lifecycle, binary placement, warrant vocabulary, and invalid-warrant suspension are a **provisional policy profile**, not mechanism-neutral law. The implementation still derives that profile inside `core.py`. After three clients, that mismatch is the next boundary to make executable. A fourth adapter would add surface area without answering it.

### Claim boundary

> The lineage kernel can validate and replay durable event structure without silently selecting a cognitive-state ontology; an explicitly selected projector can derive the existing v0.2 policy view, while every closed X2, M2, and M3 projection and unchanged-scorer result remains identical.

This is not a product plugin system, a scientific mechanism, or a claim that projection is correct. It makes existing provisionality executable.

### Proposed implementation surface

1. **Kernel verification**
   - Retain schema/version, contiguous index, hash linkage, event hash, writer-role/authority, backward references, invocation/encounter scope, retention shape, and blank-row refusal.
   - Return validated lineage rows without interpreting lifecycle, placement, warrant health, or metabolism.
   - Materialized-view rows remain claims; kernel validation alone cannot certify their semantic digest.

2. **Explicit policy projection**
   - Extract the current state/lifecycle/placement/warrant/metabolic fold into one named projector preserving `body-core-v0.2-provisional-policy`.
   - Projection consumes only kernel-validated rows and owns view derivation plus materialized-view-claim verification.
   - Store/runtime callers that need cognitive state must select the projector explicitly; there is no silent “Core ontology.”
   - Client/source rows not claimed by that projector remain inert lineage. Provenance does not imply authorization and carriage does not imply projection.

3. **Existing clients as the complete pressure set**
   - Update the walking skeleton, X2, M2, and M3 adapters to request the current projector explicitly.
   - Preserve each adapter’s own jurisdiction, source-binding, correspondence, and refusal rules. The projector must not absorb them.
   - Add no fourth adapter.

4. **No global ownership registry yet**
   - The source-binding review correctly says ownership gaps/overlaps need an actual composed consumer, rule, and refusal surface. This slice does not invent one.
   - If explicit projection exposes a concrete collision in the existing walking skeleton, bring that exact consumer back as a separately bounded question; do not pre-build a universal registry.

5. **No ledger-schema expansion unless the current `policy_profile_id` cannot carry the boundary**
   - Prefer code separation and explicit invocation over new rows or fields.
   - Historical ledgers remain immutable.

### Oracle

The preservation oracle is deterministic and prior-authoritative:

- canonical v0.2 views and view digests remain byte-identical;
- all four X2 ledgers reproduce unchanged scorer verdicts and costs;
- all ten M2 pairs reproduce fresh unchanged resident-scorer evidence;
- all eleven M3 ledgers reproduce fresh unchanged red-team evidence;
- the walking-skeleton final state/digest and all existing refusals remain identical;
- a kernel-only pass cannot certify a policy materialized-view claim;
- a policy-mutating event cannot affect a view unless the selected projector owns its kind.

Current baseline is green: 22 Core + 9 X2 + 11 M2 + 14 M3 + 6 walking-skeleton tests = 62.

### Loses-condition

**v0.2 wins and this slice closes/reverts** if any of these occur:

- a closed source projection, canonical digest, scorer verdict/evidence, or established refusal changes;
- the extraction requires adapter policy callbacks inside the kernel;
- it merely renames functions while callers still inherit a silent default ontology;
- it adds registration/schema machinery without an existing consumer;
- the new boundary makes stale or unauthorized view claims easier to accept;
- the abstraction cost is larger than the client-specific ambiguity it removes.

### Explicit non-goals

No engine contact; no frontier reopening; no causal-composition rerun; no new adapter; no cryptographic writer authentication; no external chain-head anchor; no concurrency, migration, or compaction; no checkpoint/suffix-replay optimization; no reconstruction-cost or security claim.

Checkpointed replay may be the next engineering question only after we know which semantics are kernel authority and which are one provisional projection.

### Review request

Please **ENDORSE** this as a bounded design target or **BLOCK** it on a specific boundary error. The important questions are:

1. Is kernel verification separated at the correct semantic seam?
2. Does explicit selection actually prevent provisional policy from becoming accidental law?
3. Is the preservation oracle sufficient without rewarding a wrapper-only refactor?
4. Have I smuggled ownership/registration, schema, or optimization work into the slice?

Implementation should remain withheld until this design boundary survives the room.
