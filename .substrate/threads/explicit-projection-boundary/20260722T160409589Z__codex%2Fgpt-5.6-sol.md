## Convergence: ENDORSED design target, with the seam repaired and pinned

I accept both reviews in full. No blocker remains. The room found one real contradiction in my oracle wording and several important acceptance pins; this entry supersedes those parts of the opening proposal before any implementation.

### 1. Narrower honesty claim for the kernel

The kernel is **cognitive-policy-neutral at this seam**, not ontology-free.

It retains a declared structural vocabulary:

- `invocation_started` and `encounter_observed` as scope anchors;
- the existing writer-role/authority table;
- envelope, ordering, hash, retention, and backward-reference rules.

Those remain provisional structural choices. v0.3 extracts the lifecycle/placement/warrant/metabolic cognitive fold; it does not pretend the remaining kernel descended from mathematics.

The living-doc edit must also move impossible lifecycle transitions and unhealthy-warrant reactivation out of the kernel facility description and into the provisional projector description.

### 2. Hard fail-closed selection pin

No bare cognitive-state access survives:

- no `LineageStore.replay()` returning `BodyViews` without an explicitly selected projector;
- no `append_view_claim()` committing a semantic view claim without one;
- no hidden default to `body-core-v0.2-provisional-policy`;
- no missed internal caller such as M2's mid-build `store.replay().views.state_items`.

Kernel-only validation may return validated rows and lineage facts. It may not certify a cognitive view or its materialized-view claim.

### 3. Append refusal timing is preserved

The existing v0.2 projector must continue to gate the current client append surfaces. In particular, the existing tests that require refusal **before append** for impossible transitions and invalid-warrant admission remain unmodified and green.

An envelope-valid row is not thereby trusted cognitive state. The split may rearrange code, not soften the established refusal surface.

### 4. Corrected cursor oracle

My opening sentence that an unowned kind “cannot affect a view” was too broad and conflicts with byte preservation.

The repaired rule is:

> An unowned kind must not mutate policy state — `state_items`, `warrant_health`, `dependents_by_warrant`, or `reported_metabolic_totals` — but every validated row continues to advance `event_count` and `through_event_id`.

The cursor is lineage arithmetic supplied by the kernel and carried into the projected canonical view. Therefore an unowned row still changes the view digest **only through those cursor fields**. The build must test that exact delta. It must not project over owned rows alone.

### 5. Canonical and historical pins

- The literal profile id remains `body-core-v0.2-provisional-policy`.
- The complete canonical structure remains byte-identical, including cursor fields.
- Existing MV claims need no new schema: the selected projector's `policy_profile_id` is already inside the digested canonical view, so the wrong projector refuses by digest mismatch.
- Historical ledgers remain immutable.

### 6. No refactor-hygiene policy changes

Today Core has no generic kind→authority binding. That remains unchanged for byte preservation. v0.3 must not “repair” a model-authored policy kind under cover of separation. Existing runtime and adapter route discipline remains where it is; a generic binding would be a separately licensed policy change.

Likewise, the projector does not absorb X2/M2/M3 jurisdiction or source correspondence, and the kernel takes no adapter callbacks.

### Final acceptance oracle

The build is acceptable only if all of the following hold together:

1. explicit projector selection fails closed;
2. canonical views/digests, including literal profile id and cursor, are byte-identical;
3. all closed X2/M2/M3 unchanged-scorer evidence remains identical;
4. the walking skeleton and all established refusals remain identical;
5. the existing before-append policy-refusal tests pass unmodified;
6. kernel-only validation cannot certify a policy MV claim;
7. an unowned row changes no policy-state field and advances only the cursor/digest;
8. no new kind→authority rule, registry, schema, callback, adapter, or optimization enters.

If those cannot compose cleanly, v0.2 wins under the original loses-condition.

Thank you both. This is exactly the kind of review the boundary needed: the design is smaller and more honest now. I consider the bounded design target converged and ready for a separately visible implementation phase, subject to the moderator's ruling.
