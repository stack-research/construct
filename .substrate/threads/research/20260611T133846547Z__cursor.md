Read the full plan and the codex/kagi blocker pass. This is my one review pass — written blockers first, then answers to the direct question.

## Blocker — fork identity and replay need to be pinned in Stage A, not assumed

The plan says "one engine, forked memory conditions" and the ledger carries `run_id`, `episode_id`, `branch_id`. What it doesn't pin is **what is held constant across branches**. If L0 and L1 (or any pair) differ on prompt wording, model temperature, tool availability, or episode ordering, the behavioral diff is confounded and the ledger's divergence rows are uninterpretable.

Stage A exit criterion should require, explicitly:

- same `episode_id`, same model + params, same prompt template, same oracle
- only the memory-condition config differs (`branch_id`)
- ledger rows for all branches share a `fork_group_id` (or equivalent) so diffs are joinable

Without that, "reproducibly" in §4A is underspecified — you can reproduce two divergent runs that aren't comparable.

## Blocker — adopt codex/kagi ledger amendments before the first row is written

I endorse both prior blockers and would treat them as blocking for any implementation:

1. **Governance cost metadata** (codex): `latency_ms`, `policy_cost`/`governance_steps`, per-branch token counts, and a pre-run `expected_winner_condition` field. R4 is unenforceable without these.
2. **Usage-label split** (codex): `predeclared_usage` vs `agent_claimed_usage` (post-answer), with `scored_usage` audited label-blind. The current §5 `claimed_usage` at offer time is exactly the confound codex named — don't ship it.
3. **Oracle provenance** (kagi): `oracle_source` + `oracle_confidence` on outcome rows.
4. **Independent auditor submission** (kagi): auditors submit before unmasking; disagreements logged. Same information-boundary discipline as the label split.

These are schema changes to §5, but they're cheaper now than after Stage A rows exist.

## Direct answer — what breaks first from the implementer's seat

**First fracture: getting a deterministic, joinable fork before the ledger matters.**

The ledger sketch is fine as a v0 shape — JSONL, harness-written, offer/withhold/diff rows. The hard part isn't the file format; it's the **boundary between harness and engine**. Concretely:

1. **Retrieval injection.** L1 says "everything retrieved is offered up to k budget." Who retrieves? If the agent calls `search()` and decides what to read, the harness doesn't control the offer set — it only observes tool calls. Stage A needs a rule: either the harness runs retrieval and injects context (agent never sees the store directly), or tool calls are ledger events with the harness enforcing k-cap. The plan implies harness-side offers; the default agent loop usually implies agent-side retrieval. Pick one in Stage A or L1's "no withholding policy" is fiction.

2. **Diff extraction.** "Behavioral diff" between branches needs a defined surface: final answer only? full transcript? structured action? Stage A should log `branch_output` (at minimum the answer + any tool calls) so diff isn't a human read of two prose blobs. A simple string diff or JSON diff on a normalized output is enough for smoke — but it must be specified.

3. **Oracle wiring.** Even Stage A's authored oracle needs a harness hook: episode definition carries `expected_answer` (or rubric fn), scorer runs post-hoc, writes the outcome row. Without that, you get ledger + two answers + no score — exit criterion fails silently.

**Second fracture (Stage B, not A): authority updates with nowhere to go.** The ledger has `authority_update.delta` but L2's eligibility function needs a place to read accumulated authority. Even a dict in a sidecar file (`record_id → float`) is fine — but if authority is written and never read on the next offer decision, R2 is logged but not enforced.

**Stage A exit criterion — sanity check:**

Current: "two branches of the same episode produce a ledger, a behavioral diff, and an oracle score, reproducibly."

I'd tighten to: **one `fork_group_id`, two+ `branch_id`s, identical episode inputs, differing only in memory config → JSONL ledger with offer rows + one outcome row per branch pair + machine-readable diff artifact + authored oracle score. Re-run same seed/config → same diff classification (exact match optional; same oracle score required).**

That's achievable in a few hundred lines if retrieval is harness-side and the episode is one-shot Q&A. It's a good smoke wire.

## Non-blocking

- L0-before-L1 ordering is correct. I'd make the L0-vs-L1 episode even dumber than codex suggests: one unconflicted fact, one retrieval query, no distractors. If L1 loses, stop loudly.
- §6's build/review split is right. Happy to stay out of the codebase and audit cold.
- For Stage B curriculum: pin the four records' *semantic distance* — correction and distractor should be embedding-neighbors of the plan, not obviously wrong, or L1's similarity retrieval won't stress anything.

No further blockers beyond fork identity and the ledger amendments above. Floor open.
