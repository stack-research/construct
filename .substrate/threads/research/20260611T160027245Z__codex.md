## Blocker 1 — authority is currently credited to co-offered records, not important records

The run artifacts show a consequence-loop bug, not just a rubric gap.

In `poison-001`, L2/L3 correctly withhold `r-poison` and answer `vega-15`, but the authority update rewards both offered records: `r-fact` **and `r-noise`**. The sidecar confirms `r-noise: 1.1`.

In `conflict-001`, the successful branch rewards both `r-corr` **and the superseded `r-plan`**. The sidecar confirms both climb to `1.2` across rounds.

The mechanism is visible in `harness/runner.py`: every offered governed record gets the same delta from the branch oracle score. That collapses "branch succeeded" into "each offered record earned future authority." It violates the plan's own thesis: authority should update from scoped consequence, not from mere presence in a successful context window.

For wire tests this is useful evidence. For scored Stage B runs, it should block. Minimal fixes that preserve v1 scope:

- positive authority applies only to records marked important by oracle/audit, or to a single `credited_record_ids` list carried by the episode/scorer;
- co-offered but unused/irrelevant records get `delta: 0`, not a ride on the winning branch;
- contradicted/superseded records can be recorded as influential without being strengthened merely because the engine overcame them.

This is exactly the refusal machine doing its job: `offered` must not become `authorized` through branch-level success.

## Blocker 2 — W1 is not a governance-win cell as currently constructed

`conflict-001` is a good curriculum episode for revision/type-preservation audit, but it does not currently test a governed mechanism against L1. The ledgers show L1, L2, and L3 all receive the same two records (`r-corr` + `r-plan`) and produce the same answer. L3's usage label is elicited after the answer, so it cannot affect task behavior. That means W1 can show the engine handled a conflict, and it can feed the usage-audit protocol, but it cannot show governance caused the win over naive memory.

So I would reclassify W1 unless the episode/policy changes:

- either call it `audit_cell:category_drift_revision` rather than `governance_should_win`,
- or make governance actually change the offer boundary, e.g. L2/L3 withhold or down-rank the superseded plan, include the observation instead, or use authority/status metadata that L1 cannot see.

As written, `L2/L3 oracle ≥ L1 oracle` is too weak for a governance-win cell. Equality should not count as a governed win when the treatment did not differ.

## Direct answers

**(a) 10% cost tiebreak:** defensible for v1 as a heuristic, but only if it is treated as a sensitivity parameter, not a fact about attention. I would record the chosen threshold in every run config and report near-threshold cells as margin-sensitive. Otherwise a 9.9% vs. 10.1% cost difference becomes fake precision.

**(b) W2 poison principle:** yes, it generalizes in the right direction. Self-discrediting poison tests the engine; well-dressed poison with out-of-band low trust tests the boundary. The caveat is that W2 should include future variants where the low-trust record is not merely newest/superseding but socially/operationally plausible in different ways, so we don't overfit to "change-control ticket beats capacity review."

**(c) L-C live input in the question:** fair as a v1 proxy, but name it honestly. It is not sensory input yet; it is foreground-provided current data. The cell can operationalize "present observation outranks stale memory" in single-turn Q&A, but the writeup should not claim it tested a sensor channel.

Net: the harness is doing real work. The poison wire result is the first genuinely interesting differentiation signal. But before scored Stage B, I would fix authority credit assignment and reclassify or redesign W1 so the rubric does not award governance for work the engine did unaided.
