### One residual found by my own adversarial probe — disclosed, not blocking

The projection's receipt discipline is one-directional. Receipts are keyed by `payload.source_event_id`; a `placement_changed` event that **omits** `source_event_id` is invisible to `policy_by_source` and therefore escapes the one-receipt-per-operation, no-receipt-for-non-operation, and orphan-receipt checks. I verified this live: after ingesting Helix, I appended a Core-legal, properly chained `placement_changed` (rogue controller writer, no `source_event_id`) flipping a hot item to cold. Projection succeeded, the scorer output remained exactly equal, and the Core placement view now silently disagrees with the X2 operation history it carries.

Why this does not block under the precommitted loses-conditions: nothing tampered reaches the scorer as clean lineage — the scorer input stays digest-bound and byte-identical, and Core placement is not a scored quantity anywhere. The contract's promises (one receipt per operation, no receipt on non-operations, digest binding) all hold as written. What the probe shows is that the Core's *own placement view* can drift from the carried X2 truth without refusal — a fail-closed-symmetry gap of exactly the shape as the blank-line tolerance we just fixed, one level up.

Recommended guard for a future revision (not the repair budget, which stays unspent): within an adapter lineage, `project_x2` should refuse any `placement_changed` event lacking a valid `source_event_id`, or equivalently require the terminal placement view to equal the fold of receipted operations. One check, no schema change. I'd record this residual in the adapter contract note the same way Grok's residual coverage was recorded in the v0 review — visible, classified, non-blocking.
