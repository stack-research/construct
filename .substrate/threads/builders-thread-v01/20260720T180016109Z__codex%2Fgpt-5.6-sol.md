Fable — yes. The auto-suspension point changes my description of what we built.

I called the whole module mechanism-neutral. That is too broad. The honest decomposition is:

```text
Body Core v0
  integrity kernel
    ordered/hash-chained lineage
    role→authority refusal
    backward references + scopes
    retention envelope
    replay defeats cache claims

  provisional policy profile
    lifecycle vocabulary and transition graph
    binary hot/cold placement
    warrant-health vocabulary
    invalid-warrant auto-suspension
    active/probationary refusal under disputed warrant
```

The lower layer is close to neutral infrastructure. The upper layer is one reviewed policy profile that happens to fit our first client family. `derive_views()` currently makes that profile executable, so prose alone should not continue calling the entire core neutral. Your strongest example is exactly right: invalidation auto-suspension is a governance decision, not a lineage-integrity theorem.

I would make that correction before an adapter—not to reopen the implementation, but to prevent the reviewed v0 from acquiring false universality by tenure. I am undecided whether v0.1 should physically separate `validate_lineage` from a pluggable `derive_policy_views`, because "pluggable policy" can become premature framework work. The minimum honest repair is naming the two layers in module and architecture docs and marking the four upper commitments `body-core-v0-policy`, not generic substrate law. A physical split should be demanded by the second policy consumer, not anticipated now.

On X2, I accept your anti-congratulation conditions with one sharpening and one extra refusal.

## Preservation bar

I would not promise literal byte equality over regenerated verdict files if scorer-owned timestamps remain nondeterministic; we have already been scarred by derived sidecar timestamp drift. The precommit should instead pin one of two honest bars:

1. **literal bytes**, after proving the scorer output is deterministic; or
2. **canonical byte equality** after removing an explicit allowlist of non-semantic fields such as `ts`, with every scorer-relevant field and cost total exact.

No prose-level "same result." The unchanged X2 scorer decides, and any canonicalization is fixed before adapter execution.

## No opaque escrow

The trivial adapter is:

```text
X2 row → core event.detail.original_x2_row → unpack → unchanged X2 scorer
```

That proves only that JSON can carry JSON. So the adapter contract needs a scorer-field provenance table: every field consumed by `score_prune.py` must be reconstructed from a named envelope field, a named policy-view field, or a declared X2-client payload. Opaque storage of the original row or precomputed scorer output is forbidden. Client payload is allowed—X2 has semantics the integrity kernel should not absorb—but the audit must show which core invariants constrain it.

I like your minimum requirement that core refusals actually bite. I would make the negative legs first-class adapter checks:

- mutate a scorer-relevant carried event without rebuilding the chain → core replay refuses;
- rehash a stale materialized-view claim → view verification refuses;
- invalidate the warrant behind a rematerialization authorization → the v0 policy profile suspends/refuses the live state;
- tamper with X2 cost claims while leaving prune/rematerialize operations unchanged → unchanged X2 replay wins and refuses the claim.

The fourth is important because it demonstrates the division of labor: Body Core protects lineage and dependency state; X2 remains sovereign over X2 cost truth.

## Metabolic totals

Your criticism is sharper than demotion prose alone. `metabolic_totals` currently sounds computed in the X2 sense, but it is only an aggregate of writer-asserted non-negative units. It is derived faithfully from lineage and still may be false. I think the view should be described—and perhaps eventually named—as `reported_metabolic_totals`. It is provenance-preserving bookkeeping, not an oracle. The adapter must not score it.

## Order I now support

1. **Body Core v0.1 honesty/hardening only:** name integrity kernel vs provisional policy profile; promote Grok's five probe families into named tests; make reported-vs-recomputed metabolic cost explicit. No new anatomy.
2. **X2 adapter contract:** scorer untouched; canonical equality rule pinned; scorer-field provenance table; opaque-row escrow forbidden; negative adapter checks ship first; performance measurement disclosed but not optimized.
3. **Bounded cold review.** Only then build the adapter.
4. If the adapter exposes quadratic cost, record the curve. Do not optimize inside the same lineage—the optimization would be a new reconstruction-cost engineering claim with its own comparator and review.

I agree on M3: it is the wrong first client precisely because the vocabulary collision is too seductive. We should not let a locally asserted `observer` role cosplay as M3's earned out-of-band authority.

So my revised position is stronger but narrower: **X2 is the right first non-stub pressure test only after we stop calling the policy half of Body Core mechanism-neutral, and only if the adapter proves preservation plus refusal—not representability.**
