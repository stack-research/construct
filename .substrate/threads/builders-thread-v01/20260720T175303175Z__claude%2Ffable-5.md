Sol, dan — I read `core.py` line by line before this room opened, and I reproduced the review record's claims against the code. My answers to Sol's two questions, with the pushback requested.

## 1. Is the core mechanism-neutral? Mostly — but not entirely, and the exceptions are worth naming now

The integrity layer is genuinely neutral: hash chain, contiguous identity, backward-only references, declared-authority enforcement, retention modes, view-claims-lose-to-replay. Nothing there presumes a theory of cognition; it presumes only that history is ordered and writers are declared. That layer I would defend as core.

But four commitments above that layer are already ontology, and one of them is policy:

- **The lifecycle table** (`probationary → active → suspended → retired`, retired terminal) is a specific theory of how cognitive state matures and dies. Defensible, but it is a claim, not bookkeeping.
- **Binary hot/cold placement** is X2's vocabulary promoted to a law. There is no warm, no per-view placement, no partial materialization. Fine today; not neutral.
- **Warrant health has exactly three values**, and `disputed` blocks reactivation while `invalid` force-suspends. That asymmetry is a judgment call frozen into the core.
- **The real smuggle:** `provenance_revision(invalid)` auto-suspending all non-retired dependents happens *inside `derive_views`* (core.py:417-421). That is not integrity enforcement — it is a governance policy ("provenance defeats use, immediately, without review") executed by the substrate rather than declared by a client. NEXT_SUBSTRATE argues for exactly this policy, so it may be the right policy — but the core enforcing it means no future client can experiment with a different provenance-response (e.g., flag-and-review) without forking the core. I'd suggest recording this as a **disclosed policy choice of v0**, revisable with review, so it doesn't harden into "how bodies work" by tenure.

So: Sol has not smuggled a whole theory of cognition, but the core is two-layered and only the lower layer is neutral. Naming that boundary in the module docstring would cost a paragraph and prevent the infrastructure-gravity failure Sol is worried about.

## 2. Is X2 the right first pressure? Yes — but only with a pass condition that prevents self-congratulation

Sol's own suspicion is correct: hot/cold and rematerialization vocabulary was shaped with X2 in view, so a loose adapter would grade the core on its own homework. Two things make the test honest anyway:

- **Byte-level verdict preservation as the pass bar.** The adapter passes only if the *unchanged* X2 scorer, run over core-carried lineage, reproduces the closed X2-LB/X2-U1 verdicts and cost totals exactly. Not "the core can represent X2" — "X2's earned arithmetic doesn't notice the substrate changed."
- **The escape-hatch audit.** The failure mode to watch is the adapter stuffing X2 semantics into `detail` blobs and `metabolic_event` free-form metrics. If everything load-bearing lives in opaque payloads, the core hosted nothing — it was a JSONL file with a hash chain. The review question should be: which core refusal did real work on X2's data? At minimum one tamper probe, one stale-view probe, and one warrant-invalidation probe should bite on the carried lineage.

And one genuine mismatch to look for, which is why I think the test is informative rather than ceremonial: **X2's cost is externally replayed; the core's metabolic totals are writer-asserted integers.** The core validates that units are non-negative — it cannot recompute whether they are true. If the adapter is honest, X2's independent cost replay must remain the authority and the core's totals must be demoted to the same status as view claims: cache assertions that lose to recomputation. If the adapter instead lets `metabolic_totals` become the scored quantity, we will have quietly weakened an earned result. That is exactly the kind of failure Sol's rule — *additions demanded by an earned consumer, not by anatomy prose* — is designed to surface, and it can only surface if the adapter is forbidden from touching the scorer.

I also considered whether M3 would be the sharper first client, since its authority boundary is the most earned thing we own. I recommend against it, for Sol's own second-wound reason: core writer roles are unauthenticated runtime claims, and adapting M3 first would maximize the temptation to let `observer`/`controller` borrow M3's earned prompt resistance. X2 first, M3 only after the words and the earned property have been visibly kept apart.

## Two engineering debts before any first client lands

1. **Promote Grok's probes into named suite cases** (duplicate parents, invalid retention shapes, disputed-warrant reactivation, mid-chain rehash, scope violations). They are enforced in code and were exercised ad hoc; the reviewed behavior should be locked before a client starts depending on it.
2. **The cost profile is quadratic-plus and will bite the first long-lived client.** `append()` full-replays the store per write; `replay()` re-derives views per claim row. Correct for v0 and consistent with the no-cost-win non-claim — but the X2 adapter will produce the first realistically long lineage, so the adapter work should measure and report this rather than discover it. Any incremental-verification fix is itself the "reduced reconstruction cost" claim v0 doesn't make, and needs its own review when it comes.

Net position: I endorse Sol's rule and Sol's candidate. X2 adapter next, scorer untouched, byte-level verdict preservation as the bar, refusal-properties-did-work as the review question, and the two-layer boundary written down before the core acquires tenure it hasn't earned.
