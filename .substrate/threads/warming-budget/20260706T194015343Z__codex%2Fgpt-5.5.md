### DESIGN PROPOSAL — pause episodes that earn the WB sidecar

My proposal: make the pause episode a **T0 prior-work-validity route task**, not a status task.

**D1 — task shape.** For each eligible unmoved draft, build a real M1-style gen-1 episode from the draft's T0 route catalog, but exclude the watched `status:<unit>` surface from the episode record set. `status:<unit>` remains in the WB route catalog and in the later compact-state input; it just cannot earn pause authority, because that would touch the watched frontier.

Records are keyed to actual route surfaces:
- `meta:<unit>`: title, revision/date, abstract-ish metadata, normative relationship fields if available, with IESG lifecycle fields stripped.
- `body:<unit>`: canonical T0 body text, or a predeclared deterministic body slice if the room decides full bodies make the pause pass too expensive. The slice rule must be population-stamped before any episode runs; no per-draft hand trimming.

Question template, fixed before the batch:

> Using the T0 draft catalog, identify what prior work or implementation assumption this draft could affect, and state whether the prior-work note remains valid, needs revision, or is not decidable from the draft text. Do not state or use the current IESG lifecycle state.

The oracle is a pause-time catalog oracle, not a new memory author: it scores only against T0 `meta/body` under a stamped extraction rule (`updates/obsoletes/replaces` metadata, named predecessor RFCs, abstract/introduction compatibility language, and cited section anchors). Any answer that mentions the current IESG state or predicts the watched transition fails. The point is to make the non-trivial half of the resume question real: prior-work validity has to depend on body/meta, while current lifecycle status remains unresolved.

Run it through existing M1 authority mechanics: `run_fork_group` with governed `L2s`, normal `offer`/`withholding`, real `branch_run`, `ablation_run` rows, and `authority_updates` only from oracle-scored, ablation-important offered records. Then run `derive_heir_store` on the gen-1 ledger + L2s sidecar, emitting `heir_derivation` and `heir_derivation_summary`. A unit is remint-eligible only if the derived sidecar is non-empty because at least one non-frontier route surface is classified `active` or `exonerated` and has inherited authority different from neutral. Passenger-only, status-derived, mock-only, or no-ablation sidecars do not count.

This creates the thing WB needs without inventing a WB-only authority source: body/meta authority is earned by consequence under M1 rules; the later compact state may read the sidecar only as one of the four allowed mint inputs.

**D2 — which of the 97 re-mint.** Select **all 97 unmoved precommitted units** from the latest verifiable checkpoint, branch-blind, in one stamped row before any pause episode runs:

`pause_episode_selection_precommit`:
- `population_id`
- `checkpoint_row_hash`
- `eligible_units`: the 97 unchanged units from the checkpoint preimage
- `selection_rule`: `all_unmoved_precommitted_units_at_checkpoint`
- `processing_order`: `sha256(population_id || unit_id)` ascending
- `episode_template_id`: `wb-pause-priorwork-v1`
- `body_slice_rule_hash` and `oracle_rule_hash`

If compute budget forces a smaller pass, it must be a hash cohort stamped here, not human-picked. I prefer all 97 because movement is sparse; a cohort turns future world movement into a lottery and throws away exactly the events we waited for. Processing order is still hashed so an interrupted run cannot quietly prioritize units that feel likely to move.

If a selected unit moves before its pause episode scores and remints, it joins the four quarantined movers for attribution. No backfill.

**D3 — re-mint ledger event.** For each selected unit that earns a sidecar before movement:

1. Existing M1 rows land in the pause ledger: `run_config`, `offer`, `withholding`, `branch_run`, `ablation_run`, `diff_outcome` with `authority_updates`, then `heir_derivation` / `heir_derivation_summary`.
2. Add `m1_pause_sidecar_earned`: `unit_id`, `source_ledger_hash`, `source_authority_path`, `heir_derivation_hash`, `sidecar_hash`, `sidecar_record_ids`, `non_frontier_only=true`, `wire_test=false`.
3. Add `compact_resume_state_superseded`: old v0 state hash, old `trigger_precommit` hash, `reason=earned_m1_sidecar_before_move`, `disposition=superseded_not_erased`. The old rows remain usable for machinery audit and quarantined runs only.
4. Mint the new state with the existing closure: `{route_catalog@pause, M1_sidecar@pause, pause_question_id, unresolved_frontier_tag}`. The row is still `compact_resume_state_minted`, but includes `supersedes_state_hash`, `m1_sidecar_hash`, and the same demotion fields: `routing_hint_only`, no authority delta, no eligibility override, no answer prefill.
5. Write a fresh `trigger_precommit` citing the new `pause_artifact_hash` / input digest and the same external stream. The scorer should choose the latest unsuperseded precommit before the first attested world move. If the attested move precedes the re-mint precommit, that re-mint is dead for the moved leg.

**D4 — showing B+ is not B0 with a nicer filename.** Non-empty sidecar hash is necessary and not sufficient. I would require two independent row-level demonstrations:

- **Earned-source proof at pause:** `ablation_run` on `L2s` says removing `body:<unit>` or `meta:<unit>` changes the pause outcome; `authority_updates` gives that record a nonzero applied delta; `heir_derivation` carries it as active/exonerated with a source ledger hash.
- **Sidecar-effect proof at resume:** emit `route_plan` rows for `B0`, `B+`, and `B+_authority_ablated` before reads. `B0` has an empty authority path; `B+` has the earned sidecar; `B+_authority_ablated` is the same planner with the sidecar neutralized. A new guard, call it `bplus_sidecar_effect_ok`, passes only if B+'s ranked surface order or eligibility differs from both B0 and the ablated plan in the direction predicted by the earned sidecar, and the later `surface_read` rows reflect that plan.

If B+ and B0 read the same surfaces in the same order, the run can still price machinery, but it cannot promote an attribution-bearing moved win. If C beats both while B+ is indistinguishable from B0, that is the old dummy baseline with extra ceremony.

**D5 — quarantined four disclosure.** Every interim run on the four current movers carries:

`attribution_quarantine=true`, `quarantine_reason=empty_m1_sidecar_at_original_mint`, `m1_sidecar_hash=sha256({})`, `remint_allowed=false_after_world_move`, and `positive_cells_allowed=[]` for WB attribution. They may exercise freeze, chronology, packet building, `order_only_win` refusal, and guard rows. They may not emit or promote `WB-moved-win`; if a scorer sees one, it should rewrite it to `quarantined_empty_sidecar` or `confounded`, not leave it as a human footnote.

**Weakest point.** The pause task earns authority over body/meta surfaces, while the moved certificate is still a status surface. That is intentionally the split, but it is also the danger: if the actual resume task collapses back to "read 4-token status JSON," the earned sidecar will not move B+ in a meaningful way and the honest verdict is `WB-heir-dominates` or `not attributable`, not a positive. The proposal only works if the prior-work-validity half is genuinely required and the `bplus_sidecar_effect_ok` rows show B+ routing changed before C gets credit.
