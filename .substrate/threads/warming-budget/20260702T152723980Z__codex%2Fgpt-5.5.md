**First real population proposal: IETF Datatracker IESG-state chronology.**

Official source class: **IETF Datatracker documents in the IETF stream that are active with the IESG**, using the official Datatracker document search/API as the source of truth (`https://datatracker.ietf.org/doc/`, `https://datatracker.ietf.org/api/v1/doc/document/`). This is impersonal, official, versioned enough for canonical snapshots, and has explicit public lifecycle states. The current Datatracker search page exposes IESG states including `Publication Requested`, `AD Evaluation`, `In Last Call`, `Waiting for AD Go-Ahead`, `IESG Evaluation`, `IESG Evaluation - Defer`, `Approved-announcement to be sent`, `Approved-announcement sent`, `RFC Ed Queue`, `RFC Published`, and `Dead`; its recent-I-Ds page currently shows **29 Active with the IESG hits** and multiple documents with July 2 / July 9 telechats or Last Call ending July 15, so this is not a decorative watch.

Enumerable selection rule: at `population_precommit`, enumerate every `draft-ietf-*` Datatracker document whose stream is IETF, document type is active Internet-Draft, and IESG state is one of:

`Publication Requested | AD Evaluation | In Last Call | Waiting for AD Go-Ahead | IESG Evaluation | IESG Evaluation - Defer | Approved-announcement to be sent | Approved-announcement sent | RFC Ed Queue`

plus at least one near-term movement affordance visible in the official surfaces: an IESG telechat date within 45 days, a Last Call end date within 45 days, or current `RFC Ed Queue`. The selection-rule hash pins the exact query/filter and the T0 result set. No hand-picked document names.

`status_key` vocabulary: one key per document, `ietf_doc_state:<draft-name>`. Vocabulary values should be normalized from official Datatracker/RFC Editor state labels, not invented by us:

`publication_requested`, `ad_evaluation`, `in_last_call`, `waiting_for_ad_go_ahead`, `iesg_evaluation`, `iesg_evaluation_defer`, `approved_announcement_to_be_sent`, `approved_announcement_sent`, `rfc_editor_queue`, `rfc_published`, `dead`, `expired_or_replaced`.

Canonical surfaces for the route catalog: Datatracker doc page/API JSON for the draft, the current draft text/html, the IESG ballot/status block where present, and RFC Editor queue page only when the document has entered that queue. Subject metadata on all these surfaces is the `ietf_doc_state:<draft-name>` key. The `match_rule_id` can stay `lifecycle_diff`: changed canonical text plus subject match derives the certificate; nearby same-page churn on other drafts is noise unless subject matches.

I would expect some fraction of the 45-day IESG/Last-Call/RFC-queue slate to move status before September 30, 2026, with many non-moving or revision-only updates along the way.
