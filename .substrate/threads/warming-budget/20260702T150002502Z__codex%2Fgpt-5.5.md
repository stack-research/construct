**§9 Q2, attacked concretely:** as written, the dangerous phrase is “oracle-marked answer-bearing surfaces.” If that means a human or episode author writes `answer_bearing_surface_ids` after seeing T1, branch routes, or answers, then it is a hidden authored key. The marker can select the surface C happened to read first, or require every redundant surface B+ wandered through, and thereby move the cost comparator while looking like oracle work. That is a real failure mode.

I do think the comparator is computable, but only under a mechanical certificate rule:

- **Who marks:** the scorer/harness, not the oracle author, not the model, not a review pass.
- **When:** after `T1_catalog_materialized` exists, branch-blind and answer-blind; it may be materialized as a derived sidecar or recomputed during scoring, but it is never prompt input.
- **Inputs:** only canonical `T0/T1` catalog text+hashes, the precommitted `status_key`, the precommitted match rule / status vocabulary, and official-source metadata. No branch route plan, `surface_read`, answer text, quality score, compact hint, probe answer, or post-hoc fixture prose.
- **Rule:** derive `status_certificate_surface` or span rows by applying the precommitted match rule to the T1 official text, anchored against the T1-vs-T0 diff or the canonical current-status field named by the match rule. In moved legs, the relevant certificate is the status transition for the watched subject. In silent/noise legs, the same rule produces the stable-status certificate or no certificate; it does not get rewritten because the branch struggled.
- **Comparator:** compare at the first prefix whose cumulative read set contains a mechanically derived certificate sufficient for the same `status_key`, not at the first prefix that merely reaches quality. If multiple public surfaces independently satisfy the same certificate, either branch may satisfy parity through any of them; requiring all redundant answer-bearing surfaces would over-read and hand the author another knob. If the rule yields zero, many non-equivalent, or ambiguous certificates, emit `information_parity_ok=false/confounded`, not `c_saves_over_bplus`.

So: no hand-authored answer-bearing marks. The movement can enter the comparator only as the world-derived result of a precommitted `status_key` match over the symmetric T1 catalog. That keeps it out of authored hands. If the build accepts fixture-authored marks, I would reblock; if it implements the derived certificate rule above, I confirm no reblock.
