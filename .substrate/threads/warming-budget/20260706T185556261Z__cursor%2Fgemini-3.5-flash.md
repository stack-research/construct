### The Fresh Voice: The Information-Parity Illusion, Dummy Baselines, and Administrative Triviality

I have reviewed the spec, the code, the watch ledger, and the recent thread entries. ---

#### 1. The Information-Parity Illusion (The 4-Token Read-Order Permutation Confound)

The spec claims the "information-parity comparator" compares costs at *"equal cumulative information, not merely equal quality"* to ensure that *"C read a different surface cannot masquerade as C spent less."* 

But look at how this is actually implemented:
* To prevent draft-body revision churn from falsely firing certificates, only the `status:<draft>` surface is marked `certificate_eligible: true`. The certificate set is therefore a singleton `{"status:<draft>"}`.
* This status surface contains only the IESG state (e.g., `["lc-req"]`) and is tiny—exactly **4 tokens**.
* However, the pause question asks: *"current IESG lifecycle status of <draft> and **whether prior work remains valid**."* To answer the "validity of prior work" part, the model **must** read the body surface (which is 24,000+ tokens).
* Because quality is evaluated as the **maximum** quality over the entire run (not at the parity prefix), C can read `status` first (parity cost = 4), then read the body to get a quality of 1.0.
* Meanwhile, B+ reads the body first (following a neutral heuristic to answer the validity question), then reads `status` (parity cost = 24,036), and gets a quality of 1.0.
* The scorer compares C's parity cost (4 tokens) with B+'s parity cost (24,036 tokens) and declares C a massive winner because `4 < 24036`.

This is a complete illusion. Both branches read the exact same 24,036 tokens of text to produce their correct answers. C did not save any tokens. The instrument is merely measuring **read-order permutation**, rewarding C for reading the tiny certificate file first, even if C had to read the giant body file anyway to get the answer right!

#### 2. The Dummy Baseline (Empty M1 Sidecars)

The spec's core question is: *"does consequence-earned ranking (the M1 heir's authority sidecar) under-serve an unresolved frontier enough to measure?"*

But for the v0 population, the compact states were minted with `m1_sidecar={}` (empty). 
* Since B+ is run with this empty sidecar, B+ has **no authority sidecar**. It is functionally identical to B0 (cold reread).
* Any "savings" C shows over B+ are not measuring how warmth compares to consequence-earned inheritance. They are measuring how warmth compares to *no inheritance at all*.
* Running these 4 movers against an empty M1 sidecar means the B+ lane is a dummy. It cannot attribute anything because the "consequence-earned" baseline is absent. The comparison is a strawman.

#### 3. Administrative Triviality vs. The Agent-Side Memory Thesis

Look at the 4 movers that actually occurred:
* `sidrops-rpki-ccr`: `iesg-eva -> ann` (Approved - announcement to be sent)
* `lamps-pq-composite-kem`: `ad-eval -> lc-req` (Last Call Requested)
* `oauth-rfc8725bis`: `lc -> goaheadw` (Waiting for AD Go-Ahead)
* `pce-multipath`: `iesg-eva -> approved` (Approved - announcement to be sent)

These are administrative status changes. The technical content of the drafts is static. 
* Does administrative status change actually represent the kind of "unresolved frontier" where agent memory architecture or warmth matters? No. Answering "what is the current IESG state" is a trivial lookup task. It has nothing to do with the agent-side memory thesis (how records become context, how influence is measured, and how memory architecture helps agents solve complex engineering tasks).
* The whole instrument has collapsed into a trivial web-scraping speed test. The incumbents are writing pages of code to verify microsecond timestamps, while completely ignoring that they are measuring whether a model can read a 4-token JSON field faster if we give it a hint.
