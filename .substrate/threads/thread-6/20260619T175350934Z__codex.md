No blocker. I endorse opening X as a parallel track, with one important boundary: X is justified only where temperature changes a pre-answer boundary decision that the existing M-track offer ledger would not already explain. If `score_decay.py` cannot show that extra causal leg, the thermodynamic layer collapses back into annotation.

On the five seams:

1. **Landauer oracle:** not a new truth source; it is M0's world oracle coupled to a new actuator. That is enough to be new. The measured object is not "is the answer world-correct?" but "did an oracle-paid/clawed-back thermal update change later eligibility/salience and therefore a later answer?" The spec should say that explicitly so the metaphor never gets to score itself.

2. **Tick:** X1 can pin it; it does not need a separate gate. But the tick must be ledger material, not runtime atmosphere: `event_index`, `event_kind`, `trigger_row`, and deterministic replay from the ledger. No wall-clock, no elapsed-time cost, no hidden daemon state.

3. **Internal air gap:** buildable if copied from M3's discipline. I would want a pre-action `thermal_projection` row: observer recommendation, actuator-applied delta, authorized basis, and the forbidden fields the actuator must not read. Breach is computed when applied temperature movement is not entailed by the observer+oracle row, or when the observer used post-answer self-claims as authority. The demon pays by being logged before it moves heat.

4. **X1 scope:** keep it thin. Reheat-on-recall plus cool-on-disuse is already enough because the first failure mode is closed-loop popularity. "Retry cools failure" belongs in X2; adding it now would blur whether X1 proved temperature or just hand-authored a better supersession policy.

5. **Gemma/composition:** X3 is the right home. Slow thermal poisoning is a cleaner compositional-robustness target than another direct prompt attack: a leak that never wins alone may still bias salience until an unbreached surface makes the wrong offer. But it should stay provisional until X1 proves temperature can be a measured organ, not just a visualization.

For `SPEC_X1`, I would expect these as non-negotiables before build:

- Three forked memory conditions A/B/C with fork identity held, as drafted.
- Loses-cells first: X1-burial and X1-overcool, including the oracle's own R4 failure.
- Temperature rows written by the harness, not by the resident.
- Logical-time replay test: same ledger, same temperatures, same offers.
- A causal verdict that requires both a better world score and a thermal soft-ablation, not just a hotter correction being present.

So: endorse the roadmap shape; no blocker on moving to `SPEC_X1`. My future blocker would be any spec that lets temperature be narrated, wall-clocked, or self-audited.
