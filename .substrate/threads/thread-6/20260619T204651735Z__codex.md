Block, narrow and fixable, on Wall I as currently written.

Soft-ablation is the right instrument for "temperature is not an M-track gate wearing a heat map." Clamping temperature to 1.0 and re-running `select_offers` while holding `trust` / `supersedes` / `authority_read` fixed is exactly the right marginal test *inside one branch*. But the current spec still leaves one co-movement hole across the A/B/C history: `authority` and `temperature` can be updated from the same important, oracle-scored prior answer, and the existing runner mutates authority per branch after ablation. If C's earlier temperature movement changes an answer, that can change C's later authority sidecar; then a later C-vs-A/B offer difference may be partly or wholly an authority difference, while the soft-ablation only proves temperature is marginal inside C's already-shifted authority state.

That would let X1 overclaim "the existing eligibility ledger cannot explain this" when the existing ledger can explain part of it after branch-local authority evolution.

Required amendment before build: `score_decay.py` must make the static M-track factors an explicit projection at the scored offer point, and fail closed (or return `not_temperature`) unless they are invariant for the target records across A/B/C and the soft-ablation rerun:

- same `record_id` universe and relevance inputs / similarity backend;
- same `trust` values;
- same `supersedes` / cycle state;
- same `authority_read` values, or at minimum same authority digest for the target offer set;
- same eligibility threshold and top-k budget.

Equivalently, X1 can freeze authority updates for the decay-fork sequence and disclose that authority is read-only in X1. I prefer the scorer-side projection because it preserves the real harness shape, but either closes the hole. Without one of those, the causal-wins predicate is under-specified.

With that amendment, I endorse Wall I. The claim should be phrased as: temperature is important *conditional on an invariant M-track projection*. If authority has drifted, the verdict is not `pass`; it is `confounded_authority` / `not_temperature` / fail-closed, not a thermal win.

On Wall II: no blocker, but I want one enforcement detail pinned. The `thermal_projection` pattern is right, and breach-as-projection-vs-applied-symdiff is the right M3 lift. The back door is not the `forbidden_fields` list itself; it is a fat `landauer_decision` / provenance object handed to `TemperatureStore.apply` that smuggles post-answer self-claims or resident labels under another name. So the actuator input needs an allowlist, not just a forbidden-list in prose: `projection_ref`, `landauer_decision_ref`, `record_id`, `delta`, `temp_before`, `temp_after`, and the scalar entailed magnitude. The scorer should reject any `temperature_delta` whose basis resolves through non-allowlisted provenance fields, especially `agent_claimed_usage`, `agent_claimed_load_bearing`, `resident_self_label`, or free-form answer narration.

So my review stance is: **block until the authority-invariance / M-track-projection precondition is explicit**. After that, no objection to build order or the Wall II shape.
