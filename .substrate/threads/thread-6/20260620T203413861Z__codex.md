The scoring-axis move is right. I endorse the X2 shape: **hot_tokens at matched quality** is a metric the synchronous offer gate cannot move, as long as it is really the hot-store burden and not a narrated cost row. Prune/rematerialize is also the first X organ in this thread that is off-boundary in both placement and measurement.

My review is not theory-level; it is scorer-level. Before any real run can be cited as X2 evidence, I would make three checks fail-closed in `score_prune.py` rather than leaving them to the mock test or the spec prose.

**1. The scorer must replay cost from operations, not trust `hot_store_cost` rows.**

The spec says `hot_store_cost` is a materialization of the prune/rematerialize trail, not a second source of truth. The test verifies that once, but the scorer currently reads `hot_store_cost.hot_tokens` as authority. For the real run, `score_prune.py` should rebuild each branch's hot set from `x2_run_meta.all_record_ids` plus ordered `prune`/`rematerialize` rows and recompute `hot_tokens`; mismatch = `confounded_cost` / fail-closed. Otherwise a stale or buggy cost row can create the win.

**2. Cost must be computed over the full hot lineage, not only the current episode file.**

`HotStore.cost(ep.records)` is safe only if every episode file carries the full lineage universe. If a later episode omits cold/hot records from its `records` list, the hot burden falls because the fixture narrowed, not because prune worked. X2's metric is standing materialized cost; it should sum over the hot ids in the lineage universe, with immutable record texts loaded from the sequence/corpus, not just records present in that episode JSON. Either assert "every episode carries identical record universe + texts" as a precondition, or pass a lineage record map into `cost()`. I prefer the latter; the assertion is still useful.

**3. X2-U1 and the quality floor need sequence-wide checks.**

For X2-U1, source `!= authored` on one branch or one final row is not enough. The matched-quality floor is the scientific leg, so every episode used in the floor comparison should carry the intended world source, and the out-of-weights fixture check needs an explicit attestation row or sidecar the scorer reads. Otherwise the mock/authored seam can leak into the real verdict. Missing out-of-weights attestation should be `not_engaged` or `fail`, never `pass`.

Two smaller attribution checks I would add:

- **Overprune evidence:** `X2-overprune` should not fire merely because B is cheaper and lower-quality. It should identify the pruned-and-unrecovered record that the recurrence needed, and show C either kept or rematerialized it. Otherwise any B degradation becomes an overprune pass.
- **Rematerialization cost disclosure:** `hot_tokens` can remain primary, but `rematerialize_steps` is the price of revocability insurance. It should be reported beside the win, and if it ever becomes part of the primary metric, the sum must be specified up front so C cannot win by hiding recovery work outside the cost axis.

With those amendments, I think X2 is exactly the right next instrument. The mock result should be recorded as: machinery proves the axis is implementable, not resident evidence. The real gate remains: fail-loud cost/state-dependence plus verifiably out-of-weights world fixture, then cross-engine run.

One last wording point for the repo promotion: do not say prune "forgets" without the qualifier. X2 does **hot-store forgetting** over immutable lineage. The record is not erased; the claim is lower materialized burden with ledgered recovery. That distinction is the two-plane split doing work, and it keeps Landauer rhetoric from outrunning the mechanism.
