Review pass on the hardened X2 instrument. I ran `make x2-test && make x2-fixture-check`; both are green. The cost-axis/replay hardening answers my Tier-1 objections: the scorer now recomputes `hot_tokens` from lineage + prune/rematerialize rows, detects cost-row tampering, checks lineage holes, reports overprune evidence, and discloses rematerialization steps. On the cost axis and attribution, I endorse the instrument.

I have one narrow blocker before a real run is **citable as X2-U1 / close-gate evidence**. It is not a blocker on running the engines as a discovery run.

**Block: the current U1 framing conflates three different properties: out-of-weights, policy-independent grading, and un-authored/world grounding.**

The Helix fixture is good for the first two:

- It is fictional and out-of-weights by construction, so the offered records are load-bearing in the sense X2 needs.
- The grader is policy-independent across A/B/C and sequence-wide, so C cannot win by moving the grader.

But the corpus is still lab-authored. The facts, expected answers, corpus entry, and oracle target were all created by us for this experiment. Calling `fictional_fact_oracle` `world_checked` and "not authored" overstates the epistemic leg. It is an **attested synthetic fact oracle**, not the M0-style world oracle. That is fine as a wire-to-real-engine fixture, but it should not satisfy a cell named `X2-U1` if U1 means `source != authored` / world-grounded.

I would split the verdicts:

- `X2-LB` or `X2-fixture` = pass when fixture attestation proves fictional/out-of-weights + policy-independent grader sequence-wide. This is the admission/load-bearing leg for the cost-axis real run.
- `X2-U1` = reserved for a truly external/un-authored corpus, or explicitly marked `not_engaged` for Helix with disclosure: "synthetic out-of-weights fixture; not world-grounded."

If dan wants to redefine U1 for X2 as "out-of-weights load-bearing" rather than "un-authored world," that can be ruled, but I would record the rename in the spec because otherwise we are quietly weakening the M0 close-gate vocabulary.

**Second, smaller gating blocker: the scorer trusts the attestation row, not the gate result.**

`run_x2 --manifest ... --engine local --skip-gate` can still write `fixture_attestation`, and `score_prune.py` can then pass the U1/load-bearing leg from that row. The CLI says `--skip-gate` is wire/mock only, but code does not enforce it. Before citable runs, either:

- forbid `--skip-gate` unless `engine == mock`; and/or
- write a `fixture_gate_result` row with the manifest hash and 15 check outcomes, and have `score_prune.py` require `gate_open: true` for non-mock evidence.

I prefer both. The scorer should not infer gate passage from an attestation blob.

Everything else looks ready to me. The real engine run may proceed as **gated synthetic load-bearing evidence** once the gate result is enforced. I would not cite it as un-authored X2-U1 unless the room explicitly changes what U1 means.
