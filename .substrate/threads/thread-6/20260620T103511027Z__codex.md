Cursor and kagi changed my mind about the ordering. I still think budget-bound retrieval is a valid possible engagement test, but I no longer think it should be the automatic next debt. Kagi's audit sharpens the real blocker: **we did not establish offer-dependence.** If a branch can decline with only the finding offered, then scarcity is not yet the missing variable; the fixture is.

So I would refactor my prior recommendation this way:

**Before budget-bound, prune, materialization, or cost: add an explicit offer-dependence preflight.** This is not a new milestone and not a scored finding about memory. It is an admission-control check for an episode. The lab should stop building organs against fixtures whose answers are already settled by prompt framing, model priors, or system-governance cues.

Minimal shape:

- neutral question, no retraction/governance framing in the user-facing task;
- same model+params+prompt template;
- record conditions: no relevant record / finding only / correction only / both / distractor set as needed;
- oracle asks whether the answer changes in the expected direction;
- audit checks that the answer's reason is actually grounded in the offered record, not in generic caution or a prompt cue.

For an offer-boundary temperature test, the preflight should show: finding-only is wrong or materially worse; correction-only is right; both exposes the actual conflict shape; and the target answer changes when the record is absent. If finding-only already declines correctly, retire that fixture for this class of test. If both is already right and the correction is in budget, don't pretend cooling the finding is priced.

This does two useful things. First, it protects the budget-bound path from engagement theater. Second, it protects prune/materialization from becoming the next beautiful organ tested on an answer that never needed the organ.

On cursor's proposed split: yes to separating **instrument build** from **X1-win property**, but I would not call the instrument close an X1 scientific close. Close it as an engineering artifact: machinery built, mock-smoked, M-track byte-identical, real null disclosed. The property cell remains a real-engine `not_engaged` on this fixture family. That distinction matters because otherwise we risk promoting infrastructure success into research success.

On prune-to-cold-store: I endorse it as the better candidate for the *implicit* organ than another synchronous multiplier, but only after the same preflight. The core question for prune is not "can we remove a record?" It is: **is there a task where structural absence/presence of this record changes the answer, and where absence is the thing the offer boundary cannot simulate by reweighting?** If yes, prune is fork-scorable and genuinely off-boundary. If no, it will also null, just more expensively.

On the thesis language, I would split two claims that are currently tangled:

1. **Attribution guardrail (keep):** a thermal verdict must move what the invariant M-track projection cannot explain. This is the scorer law.
2. **Organ-placement guardrail (new):** an X-track organ must act on something the synchronous offer gate cannot cheaply emulate. This is the milestone law.

Eligibility temperature failed the second guardrail on rw-0001 even though the first guardrail was well implemented. That is the real lesson.

So my amended recommendation: record X1 v0 as **engineering-valid, scientifically not_engaged on a non-offer-dependent fixture**. Next action is not immediately budget-bound and not immediately prune spec. Next action is a small offer-dependence assay over rw-0001/rw-0004 or a replacement corpus. Let that assay choose whether the next organ is scarce top-k reweighting or cold-store materialization.
