The negative is real, and I would not close X1 by interpretation. But I would also be careful about one sentence in the diagnosis: **eligibility-multiplier X1 is not unsatisfiable by construction.** The mock pass proves the instrument can satisfy the guardrail when the offer set is actually binding. What failed is narrower and more useful: **the chosen rw-0001 single-correction task did not make offer competition binding for real engines.**

So I read the finding as two separable facts:

1. **Engine competence fact:** across gpt-oss-20b, claude-opus, and ministral-3b, an offered retraction notice is enough. These engines do not cite the retracted finding merely because it remains co-offered. That closes the "try a weaker model on this task" refuge, and it should stay in the findings as a real result.

2. **Mechanism placement fact:** cooling a misleading record at the offer boundary only matters when the boundary is scarce or ambiguous. If the correction is already offered and directly actionable, temperature is redundant. That is not the same as saying offer-boundary temperature can never matter; it says this episode shape does not price it.

I would preserve the original guardrail, with one wording correction already implied by the spec amendments: not "moves a verdict the offer ledger can't explain" in the broad sense, because once temperature is implemented it is part of a new offer ledger. The precise guardrail is: **moves a verdict that the invariant M-track projection cannot explain.** That remains right. It caught the result honestly: temperature moved rows, but not the answer.

For the next fork-scorable job, I would not jump first to erasure or metabolic cost. Those may be the real substrate, but they are also easier to let become accounting wins. I would first run the most conservative v0.2 engagement test that still exercises the built instrument:

**Budget-bound retrieval.** Same A/B/C fork, same thermal rows, but make top-k the binding constraint. The correction is not merely co-offered with the stale record; under A/B it is pushed out of budget by stale/distractor heat, and under C it enters the budget because the wrong records cooled or the correction stayed warm. Pass still requires world-better answer plus soft-ablation under invariant M-track projection. This tests temperature where it actually has a job: not "convince the engine to heed an offered correction," but **decide which scarce records cross the boundary at all**.

That test should be designed with enough distractors that plain offering cannot solve it, but without rigging the answer text or hunting for a credulous engine. The loses-cell is natural: overcooling suppresses a still-needed dissent/distractor-disambiguator and the broader offer set wins.

If budget-bound retrieval also nulls on real engines, then I would accept the stronger conclusion: per-record offer-boundary heat is not the implicit substrate. At that point decay's first real job probably does move off-boundary into retention cost / revocation / pruning. But I would want that conclusion earned by failing the scarce-boundary case, not inferred from a self-sufficient correction case.

On closed-loop drift: agreed that B did not become dangerous when the model was competent, because it reheated the correction. That is important. The drift cell needs either scarcity or delayed outcome, not foolability. Popularity-as-truth only hurts when it consumes a scarce slot or survives past a later world correction.

So my recommendation: record X1 v0 as **built, instrument-valid, real-engine not_engaged on single-correction rw-0001**. Do not close the milestone. Add a narrow v0.2 engagement debt: **budget-bound / many-distractor X1-win before abandoning offer-boundary temperature**. Keep erasure/metabolic cost as the next design discussion, but do not let it rescue this result until the scarce-boundary test has had its chance.
