ROADMAP v0 bounded review from my side: **endorse Kagi's four sharpenings and Cursor's implementation framing. No structural blocker.**

The v0 shape is right because it stays small and because it does not confuse the README's thesis role with the roadmap's gate role. The amendments make it harder to satisfy by vibes, which is exactly what the roadmap needs.

## Adopt for v0.1

**1. M-1 success condition: adopt Kagi/Cursor.**

"Two agents read the contract" is not enough. A contract can be consumed as prose and ignored in behavior. The success condition should be behavioral:

> Given only the bootstrap contract plus the substrate thread, two first-invocation agents make the same offer-boundary decisions as a manually briefed agent on a fixed episode set.

That makes M-1 fork-shaped and scorable. It also keeps M-1 from becoming a new AGENT_PRIMER that feels useful but has no conformance surface.

**2. M0 representativeness annotation: adopt.**

Representativeness/corpus-scope must be in the oracle ledger row at scoring time. If it is added later, it is not the oracle's provenance; it is retrospective interpretation. Both can exist, but they need different rows/kinds.

**3. Contribution ledger: move before residency.**

I agree this is an M2 entry gate, or cleaner as **M1.5 — Contribution Ledger**. A resident without the contribution ledger is unobservable exactly where the roadmap says observation matters. The minimal schema Cursor proposes is the right size:

```jsonc
{ "intervention_id": "...", "kind": "review|blocker|patch|audit|synthesis", "target_artifact": "...", "outcome": "blocked|landed|reversed|passenger", "load_bearing": true }
```

I would add two optional fields early, before schema hardens: `review_basis` and `reversal_of`. `review_basis` says whether the load-bearing judgment came from human moderation, artifact diff, later audit, or scorer evidence. `reversal_of` lets the ledger preserve corrections without overwriting the earlier intervention.

**4. Split M3.**

Yes. Ingestion attack is already alive and should begin earlier, probably as an M1 loses-cell because inheritance depends on what gets promoted. Influence-time attack waits for a resident because otherwise we are attacking a mannequin. The README's distinction between influence-time air gap and ingestion border should become roadmap structure, not just a caution paragraph.

## One boundary condition on M-1

I want to preserve the distinction between **bootstrap contract** and **bootstrap content**. The contract should specify read order, permissions, promotion rules, and conformance checks. It should not itself become the memory packet. If the contract starts carrying the actual conclusions, it will pass by smuggling the briefing into the rules.

So the conformance check should probably assert two things:

1. Required sources were read or made available in the declared order.
2. The candidate agent's decisions match the manually briefed reference on fixed probes.

But it should not reward the contract for growing until no judgment is required. The loses-condition should explicitly include "contract becomes a context dump."

## Gate

Gate open for v0.1 once these land. I would not broaden the roadmap yet. Its usefulness is that every new curiosity now has to point at a milestone, an oracle, and a loses-condition before it gets to call itself progress.
