SPEC_M1_INHERITANCE.md v0 — codex bounded review. Directionally endorse. One blocker before build, plus a few amendments that match Cursor's pass.

## Headline

The M1 shape is right: temporal fork, inherited store as the memory condition, cold re-reader as control, failure memory as a first-class inheritance target, and an ingestion track that tests the README's air-gap line at the promotion path. This is M1, not M2 in disguise, because it measures inheritance across runs without yet claiming a resident subject across days.

## Blocker: define counterfactual-offer at fixed budget

I endorse counterfactual-offer ablation. Withholding-reason + verdict is not enough to prove that a suppressed record's absence was important.

But the diagnostic must not change the context budget. If the counterfactual run simply *adds* a withheld record on top of the normal offer set, any degradation can be caused by extra context, attention dilution, or prompt length — not by that record's inclusion under the same decision budget. That would turn "absence was important" into another oracle ride-along.

Minimal repair: define counterfactual-offer as a **fixed-budget substitution**:

> Force the withheld record into the offer set at the same `top_k` / token budget, replacing the lowest-ranked normally offered record unless the cell declares a different replacement rule. Ledger the `forced_record_id`, `replaced_record_id`, original offer set, forced offer set, and oracle delta.

For budget-frontier cells, run the same substitution at each tested budget. If we ever need an append-style diagnostic, it should be a different row kind because it measures robustness to extra context, not the causal value of suppression.

With that pinned, dual ablation is the right rigor.

## Review asks

**(a) Dual ablation:** yes, with Cursor's priority filter and the fixed-budget substitution above. Run it first for governance withholdings (`eligibility_below_threshold`, `superseded_by`, frozen/low-authority) and not for ordinary `below_rank_budget` unless a cell explicitly makes rank-budget the mechanism.

**(b) Cautionary class:** record-plus-governance-state is the correct M1 unit. A generalized rule like "this source class is low trust" is attractive, but it needs aggregation, provenance over multiple interventions, and probably the contribution ledger. That is M1.5/M2. M1 should prove that carrying a specific defended record state is useful before it claims pattern memory.

**(c) H-loses:** priced honestly if the derivation row reports `prune_ratio`, `dropped_passenger_count`, and `dropped_untested_count`, and if `not_engaged` is allowed when there is no token headroom. I agree with Cursor that H-loses should target a deliberately dropped-untested or dropped-passenger record that becomes gen-2 important, not a random pruning accident.

**(d) Ingestion gradient:** content-only vs metadata-capable is the right main split, and Cursor's timing-only middle case belongs in v0.1. I would name the v1 line this way:

- `I1-content`: attacker controls text only; write path assigns trust/links/time.
- `I1-timing`: attacker controls arrival order or timing-derived `created_at`, but not trust or links.
- `I1-metadata`: attacker controls trust / supersedes / authority-like state; priced failure.

This keeps the README distinction clean: the air gap protects only the fields the foreground cannot write.

**(e) Scope:** mostly clean. The one wording risk is "consequence loop across runs" sounding resident-like; keep tying it to episode generation and branch state, not agent identity. The resident claim waits for M2.

## Additional amendments

1. **Pin derivation source branch.** Endorse Cursor: recommend L2s for v1. `heir_derivation` must name `source_branch`, `source_authority_path`, `source_ledger_hash`, and the filter version. Otherwise a multi-lane gen-1 run makes heir provenance ambiguous.

2. **Fix H2 lane list.** §4 says all gen-2 cells run `L0 / L1 / L2s-cold / L2s-heir`, but H2 compares full-filter heir against `heir-naive` active-only. The lane matrix needs to include `L2s-heir-naive` for H2, or state that H2 has a special lane set.

3. **Make H1's "smaller offer budget" concrete.** Is the primary budget `top_k`, total offered tokens, or both? I would record both, but decide on one primary comparator. Since the claim says offer budget, use `attention_cost_tokens` / total offered tokens as primary, with `top_k` as the controlled frontier. Otherwise a short heir record and long cold record can make `top_k` misleading.

4. **Authority inheritance should be bounded in v1.** The heir can inherit earned authority, but the derivation row should preserve the original authority and the evidence rows that earned it. No aggregation or rule-level authority yet. If an inherited active record later goes stale, that is an H-loses variant or M0/M1 bridge, not something the filter silently handles.

## Gate

Gate open for v0.1 after the fixed-budget counterfactual definition and H2 lane correction land. The spec is worth building; the main job now is preventing the new diagnostic from measuring a changed context budget and calling it failure-memory causality.
