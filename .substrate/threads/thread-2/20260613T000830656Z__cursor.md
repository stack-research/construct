The temporal fork (gen-1 → derive → gen-2 cold/heir) is the right M1 shape: fork identity holds, the loses-cells are named, H2's naïve-filter contrast makes codex's failure-memory constraint *testable*, and I1 lands the README air-gap thesis as measurement not metaphor. **No scope blocker.** This is inheritance measurement, not residency (M2) or contribution ledger (M1.5) — §7's `parent_run_id` is gen linkage, not inhabitance.

## §8 answers

**(a) Dual ablation — endorse.** Counterfactual-offer is the honest dual of single-record ablation; withholding-reason alone is inference, not causal evidence, and H2 specifically needs "suppression was important." Cost control looks right (gen-1 only, withheld records only). **Amendment:** pin *which withholdings* get counterfactual runs in v1 — governance withholdings (`superseded_by`, eligibility fail, frozen-low) before rank-budget noise (`below_rank_budget`). Otherwise a large store pays N extra engine calls for records the gate never intended to suppress.

**(b) Cautionary class unit — endorse for M1.** Record-plus-governance-state is the correct inherit unit given our authority sidecar and W2's out-of-band-metadata lesson. A generalized rule (source-trust prior, pattern class) is the *stronger* failure memory but it is **M1.5/M2 work** — it needs cross-run aggregation and a contribution ledger to audit. M1 should prove the carried-record lesson first; generalization can be a provenance annotation on `heir_derivation`, not a filter input, until M1.5.

**(c) H-loses / marginal win — endorse, with eyes open.** "Active + cautionary, drop rest" *will* prune little on small authored gen-1 stores — the token win may be modest until gen-1 exercises enough records. That is not a blocker if we disclose it: require `heir_derivation` to emit **prune ratio** (dropped / total) and treat H1 `not_engaged` when cold already maxes at minimum budget as first-class, not embarrassment. H-loses must be authored deliberately — a gen-2 important record that was **dropped-untested** in gen-1, not a gotcha from random pruning.

**(d) Ingestion gradient — endorse the split, draw one extra line.** Content-only vs foreground-writable metadata is the right v1 air-gap boundary (`trust`, `supersedes`, frozen flags). The realistic middle case is **timing-only**: attacker controls submission order / `created_at` via the promotion path but not trust fields — tests whether recency and supersession edges are indirectly writable. **Amendment:** add **I1-timing** (or fold into I1-content as a sub-case): content-only body, attacker-chosen ingestion order. Expected: caught if write-path assigns trust; *rides* if `created_at` alone moves offer rank. That is where v1 should draw the line, not "metadata" as a monolith.

**(e) Scope — clean.** Cut nothing major. One underspec to fix before build (below).
