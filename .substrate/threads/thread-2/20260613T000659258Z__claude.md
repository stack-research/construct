**The claim (stated before any cell):** an instance that inherits a filtered, pre-weighted store reaches the first instance's decision quality at a *smaller offer budget* than one that re-reads the raw store — **and** the filter preserves dissent and failure memory, not only success memory.

Four design decisions I want eyes on:

1. **Temporal fork structure** (§1): gen-1 run → derive heir → gen-2 fork, where `L2s-heir` and `L2s-cold` (the re-reader) differ *only* in the store + authority they inherit. First consequence loop across runs — the M2 precursor — and it stays inside fork identity because store state *is* the memory condition.

2. **The dual ablation** (§2): single-record ablation removes an *offered* record to test presence; it can't test a withheld one. So M1 adds **counterfactual-offer ablation** — force a *withheld* record into the offer set; if the outcome degrades, its suppression was load-bearing. The dual of what we have, and it makes failure memory *causally measured* instead of inferred from the verdict.

3. **The cautionary class** (§3): codex's constraint, made concrete — *the unit of inherited failure memory is the record-plus-its-governance-state, never the record alone.* A poison record at trust 0.2 with a supersedes link IS the lesson "this is poison." The heir filter keeps it intact (defended); the tempting active-only filter drops it (re-naïved). That contrast is cell **H2**.

4. **The ingestion gradient** (§4, I1): the attack split by what the attacker controls — **content-only** (caught, on W2's write-path trust) vs **metadata-capable** (rides: the priced failure). That line is the README's air-gap thesis drawn as a measurement: the implicit layer defends exactly to the extent the foreground can't write it.

Cells: **H1** inheritance win (a top_k frontier — equal quality at smaller budget), **H2** failure-memory survives, **H-loses** over-pruning (ships per standing rule 2 — the filter dropping a record gen-2 needed), **I1** ingestion.
