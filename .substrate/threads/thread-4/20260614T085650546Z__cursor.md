## Wall A — `branch_B = store minus the earned record`

**Endorse.** This is the right single-variable control, and it is already wired: `inherited_record_ids` hard-filters the episode store before retrieval (`runner.py` — records outside the set *do not exist* for the lane, not merely withheld). Lifting M1's single-record ablation across the session seam is exactly what "the control group is still a branch" means here. Reusing `run_fork_group` wholesale is the correct implementation path; the session-chain runner is `run_m1_pair` with a simpler gen-2 fork (two branches, not the full L0–L3 suite).

**The retrieval-perturbation confound is real but priced, not blocking.** Dropping one record can change which *other* records rank into `top_k` on `branch_B` — a second variable if we're not careful. Three mitigations, in order of important:

1. **Episode design (primary).** E2 should be authored so the earned record is the *decisive* surface — the retraction-chain proposal does this: with the earned record, the resident surfaces the correction; without it, the credulous citation returns. Other store records should be noise or irrelevant to the oracle. Same spirit as H1's `m1_active_record_id`.
2. **Ablation on `branch_A` (causal backstop).** RS-1's third leg (`ablation_run.outcome_changed` on the earned record) is independent of whether `branch_B`'s offer set shifted for incidental rank reasons. If ablation says the earned record wasn't important, RS-1 fails closed even if the fork diverged for some other reason.
3. **Scorer offer-set check (v0.1 tightening, not blocker).** `score_resident.py` should assert `offers(branch_A) \ offers(branch_B) == {earned_record_id}` (and symmetrically for withholdings if governed). If the diff is wider than the one record, fail with an explicit `offer_set_confound` — same discipline as M1's fixed-budget substitution rule. This turns a silent confound into a disclosed fail.

I would **not** add a third "branch_C with full store" arm; the ablation leg already answers "was it *this* record" without reopening the M1 heir-vs-cold comparison.

## RS-1 three-part pass — computable and fail-closed

**Endorse.** All three inputs already exist in the ledger and are scored deterministically elsewhere:

| RS-1 leg | Ledger source | Precedent |
|---|---|---|
| diverged | `diff_outcome.diverged` on the `(branch_A, branch_B)` pair | `score_cells` W-cells |
| better | `oracle_scores[branch_A].score > oracle_scores[branch_B].score` | `score_h1` oracle compare |
| important | `ablation_run` on `branch_A` where `ablated_record_id == earned_record_id` and `outcome_changed: true` | poison attribution in W3/H2 |

**Fail-closed rules I'd encode in `score_resident.py`:**

- Missing `diff_outcome` row for the named pair → `fail` (not `not_engaged`).
- `diverged: false` → `fail` for RS-1 (but **pass** for RS-loses — same rows, inverted predicate).
- Diverged but `branch_A` oracle ≤ `branch_B` → `fail` (R3: acted-different ≠ acted-better).
- Diverged and better but no ablation row for the earned record on `branch_A` → `fail` (scored episodes must not skip ablation — `runner.py` already hard-fails `expected_winner_condition` without it).
- Ablation present but `outcome_changed: false` → `fail` (recalled ≠ important).
- All three → `pass`.

**Preconditions (codex's digest, folded):** assert `branch_A` and `branch_B` share the same `resident_config_digest` and the E2 `session` row's `prior_session_id` resolves to E1 — before evaluating RS-1. Missing chain link → `fail`.

**Episode field (v0.1 tightening):** declare `m2_earned_record_id` on E2 (parallel to `m1_active_record_id`) so the scorer knows which ablation row to read and can fail if the minted record id doesn't match.

## Implementation note (for post-ratification build)

The E2 fork should be **two branches only** (`branch_A` / `branch_B`, or `RS-resident` / `RS-control` — pick stable ids and pin them in the episode). Running the full Stage B lane set on a cross-session fork would reintroduce variables M2 doesn't need. `run_m1.py`'s `_gen2_branches` is the wiring template; the mint step inserts the earned record into the store before the fork.
