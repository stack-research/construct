### §9c-1 Retraction-chain as legitimate un-authored leg — endorsed, one gap to close

Reusing M0's rw-0003 corpus as the resident's own earned failure is the right move. The corpus is fixed and world-grounded; it cannot be edited retroactively. Wall B ensures the harness mints the earned record, so the borrow is clean: the oracle's authority flows from the original retraction event, not from M2's episode authorship.

One gap: **RS-U1's world-grounding is currently transitive in a way the spec doesn't make explicit.** The chain is E1 oracle scores wrong (`source: retraction_corpus`) → harness mints (`mint_basis: world_correction`) → E2 fork diverges → `branch_A` scores better. "Better against the world" holds only if the **E2 oracle** also carries `source != authored` independently. If E2's oracle is authored (e.g. checking whether the resident reproduces the retraction text), the world checked E1, but *we* checked E2 — the W1' trap re-enters one level up.

**Tightening 1:** assert `oracle.source != authored` on the E2 `diff_outcome` row as an RS-U1 precondition in `score_resident.py`. The retraction-chain corpus already satisfies this (E2 evaluates whether the cited finding is current — same oracle machinery as rw-0003 in M0/HU1). The spec should state it explicitly rather than letting it ride on the corpus proposal.

### §9c-2 `corpus_scope` disclosure — honest, one addition

"Single-hop / single-chain" correctly names the hop-count narrowness, but the narrowness has two dimensions: one hop (S1→S2) *and* one retraction (rw-0003 only). A multi-retraction corpus at one hop is a different epistemic claim; a reader of the verdict ledger should not have to chase the episode file to know which.

**Tightening 2:** `corpus_scope` on RS-U1 verdicts reads `"one_hop: S1→S2; one_retraction: rw-0003"` — both dimensions named in the verdict itself, so the compounding and breadth debt is legible in the ledger. Parallel to M1.5's CB-U1 annotation, explicit on both axes.

### §9c-3 RS-stale's "reinstated finding" — authored reinstatement sufficient, but must be tagged

grok surfaced this: **yes, an authored reinstatement is sufficient for RS-stale — but it must be tagged as such.**

RS-stale's purpose is R4 (governance-loses): does the store yield to fresher foreground input? The cell's claim is a **mechanism claim** — the store can be overruled; live input outranks stale memory when the harness says so pre-run. It is not a claim the world actually reinstated anything. An authored reinstatement datum is the right oracle for a mechanism cell: it tests the yield gate, not world state.

What would be wrong is presenting RS-stale as a world-grounded claim. That requires a world-checked oracle, but RS-stale doesn't make that claim — RS-U1 does.

**Tightening 3:** add `oracle_basis: "authored_reinstatement"` (not `world_checked`) to RS-stale's scored row, immutable at scoring time — same disclosure discipline M0 applied to corpus annotations. This keeps RS-stale honest about what it tests (mechanism) without inflating it into a world-grounded result it doesn't earn. A future RS-U2 can world-check the reinstatement; not a blocker here.
