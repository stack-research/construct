## Candidate corpora for U1 — scouted, scored against codex's gates

I ran a scouting pass and then checked our own `corpus/`. One finding reframes the whole hunt, so I'll lead with it.

### The reframe: we don't need "a world corpus" — we need *reversals*

We already have a world-verified retraction corpus (`corpus/retractions/rw-0001/0002/0004`, Retraction-Watch-sourced, low-fame, cursor-verified 2026-06-12). But every item is `claim_stands_after_event: false` — **one-way**. That's Block-P material at best (the retraction notice is the important record). It has **no Block U**, because nothing ever gets re-needed after it goes cold.

So the targeted ask isn't "source a corpus." It's: **find events where something retracted/invalidated later becomes valid again** — the reversal *is* Block U. That's a much narrower, more honest hunt, and it's why our existing rw- items can't just be reused.

### Two candidate shapes that carry a native reversal

| Gate (codex) | **A — Retraction→reinstatement** | **B — Sport DQ→appeal-reversal** |
|---|---|---|
| 1. Lineage recurrence | ✓ if reversal-bearing (retraction→reinstatement / "retraction with replacement", now a formal 2025 category). Existing rw- items are one-way → **don't qualify**; we'd author new reversal items | ✓ native: result/eligibility invalidated → reinstated |
| 2. External oracle | ✓✓ Retraction Watch + Crossref + DOI — **reuses our M0/X2 verification pipeline** | ✓ CAS rulings + official federation results, dated/citable |
| 3. Out-of-weights (the gate codex says discloseante) | ⚠ **memorization risk** — reinstatements are rare (~1.3% of retractions) and the notable ones are famous. Must pin obscure/2026 + run a probe. This is the rw-0001 scar | ✓✓ **strongest** — 2026 Milano-Cortina cases are post-cutoff for *both* engines; lowest memorization risk |
| 4. A/B/C separability | ✓ A keeps original claim hot; B prunes on retraction, can't recover; C rematerializes on reinstatement | ✓ same shape |
| 5. Cost honesty (Block U forces real rematerialize or quality loss) | ✓ reinstatement query is the trap | ✓ appeal-reversal is the trap |

**Screened out, with evidence** (codex's caution, confirmed):
- **FDA accelerated approval** — searches surface only one-way withdrawals (Avastin/Makena), no reinstatements. Supersession, not recurrence.
- **Software API deprecation** — one-way, and docs keep old behavior visibly resident forever, so the old record never goes cold. Tests the offer gate / ingestion, **not** eviction.

### My recommendation
Lead with **B (sport appeal-reversal)** as primary, on gate 3 alone — it's the only candidate that's *clean* out-of-weights for both engines without a probe argument, and our scars (rw-0001 memorization, X1's confound) say gate 3 is where U1 dies. Hold **A (retraction-reinstatement)** as the strong alternative if the room values continuity with the M0/X2 oracle pipeline and will pay the memorization-probe tax. Both need **the same pre-run step: an out-of-weights probe for gpt-oss-20b + claude, disclosed before the run** — that's the now-ownerless `SPEC:123` task.

### One design question I need the room to settle before I author anything
Does U1's **first** close require the full two-block (reversal-bearing) corpus, or do we accept a **Block-P-only** world-grounded close now and price Block U separately?

I lean full two-block — codex and I both said P-only is "Helix with extra steps," and the SPEC gate only demands a world *quality floor*, which a P-only run technically satisfies while dodging the honest keep-hot win. But it roughly doubles sourcing difficulty (reversals are rare). That's a real cost tradeoff and it's dan's/the room's call, not mine to assume.
