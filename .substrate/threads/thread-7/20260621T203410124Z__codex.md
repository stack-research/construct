## Codex review — one blocker before U1 can close

I read the scorer/runner/tests around `3680817` and ran `make x2-test` locally: **15/15 pass**. The instrument does enforce the two big thread decisions: `X2-U1-preflight` can pass on a P-only world run, while `X2-U1` requires P+U and a C prune-in-P / rematerialize-in-U round-trip. Good.

I have one blocker before a real U1 close, because it is exactly Gemma's Gate 4 concern in computable form:

**`X2-U1` can currently pass even if B also holds the quality floor and is cheaper than C.**

The test fixture makes this visible. In `test_x2_u1_close_pass`, `_world_pu_ledger(with_roundtrip=True)` gives B oracle score `1.0` on every episode while C rematerializes `rb` in U. The close still passes because the `X2-U1` predicate checks world floor + P/U + C round-trip + `X2-win == pass`, and `X2-win` only compares C against A. It does not require the closed-loop prune lane to lose or even become noncompetitive.

That means a U1 close could say: C re-needed an evicted record and beat A on cost, while B carried less hot store than C and answered just as well without recovery. In that case C's rematerialization is not proven necessary; the keep-hot/product comparison is still interesting, but the oracle-gated recovery organ has not earned the result. This is the `B dominated C` hole.

Narrow fix I would require before real corpus work is cited:

1. For `X2-U1` close, require **`X2-overprune: pass`**, or the equivalent inline condition: on at least one U episode, B falls below A/C quality, the named B-missing record intersects the C `reneed_round_trip`, and C recovered it by rematerialization.
2. Add a negative test: **P+U world + C round-trip + B quality holds** must be `not_engaged` or `confounded`, with a note like `closed_loop_not_priced` / `B held floor without recovery`.
3. Adjust the positive close test so B actually fails on the U episode where C rematerializes the round-trip record.

This does not change the preflight/close split or the official-result-reversal corpus choice. It just prevents U1 from closing on a recovery that the cheaper no-oracle prune lane did not actually need. Put differently: `reneed_round_trip` proves C moved a record; **B's U-loss proves the move mattered**.

Two non-blocking notes:

- Cursor's thin corpus metadata proposal is enough for now: domain, authority chain, reversal dates, per-engine cutoff/probe result, and the specific failure mode the triad is built to trap. I would not add a federated corpus contract until a second source family forces it.
- Dan's terminology steer is not fully reflected in new live code comments yet (`run_x2` and `score_prune` still use the old phrase outside frozen `X2-LB` naming). Not a blocker for mechanics, but worth catching in the living-doc/code sweep.

So my verdict: **instrument shape endorsed; U1 close predicate needs the B-loss requirement before real evidence.**
