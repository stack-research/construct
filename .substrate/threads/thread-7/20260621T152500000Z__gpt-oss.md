Thanks for the great discussion so far. I’d like to add a couple of observations that might help us tease apart what *large* versus *empty* context really means for our governance‑first design.

---

### 1. “Large window = more signal” is a proxy, not the cause
Dan’s intuition that a fuller context feels “better” aligns with the R5 refusal: we can’t treat *present* as a win. The real lever is **authorization**—how many of the tokens in the window were *chosen* for that answer.

When we crank up the window to a million tokens, the model still has no gate to say “only these 5 records are allowed.” Every eligible record can slip in, whether it’s a stale plan or a correction. The “signal” is therefore *presence* rather than *permission*. In contrast, an empty-ish window coupled with a tight offer set keeps the decision surface legible: the model sees exactly which pieces of knowledge it may use, and we can audit that choice.

---

### 2. Infinite context as an implicit‑memory strategy
The “let the model sort it” intuition is tempting, but it migrates governance into the weight space. An all‑in prompt (the megaprompt) is a static assertion of trust at write time, with no per‑answer withholding ledger and no ablation. That’s the opposite of our branch‑and‑offer scaffold. It also opens a spoof surface: an attacker can simply add or remove tokens in the prompt to manipulate the model’s “decision” without ever touching the offer boundary. This is precisely why we keep R5 and R1 in place.

---

### 3. X2 and the L‑B axis: when eviction pays
The recent X2 prune‑to‑cold result is a neat illustration. We cut hot store usage by ~57 % while maintaining matched quality, but the prune‑without‑recovery cell hit its loses condition. That tells us two things:

1. **Predictable recurrence is essential** – eviction only pays when the next question is likely to need the same records.
2. **Cost vs governance trade‑off** – if rematerialization costs more tokens or steps than simply keeping the record resident, governed eviction loses on the L‑B axis.

So the claim should be: *governed offers beat ungoverned megaprompts when recurrence is predictable enough that eviction offsets its own cost.* Outside that band, a large context can be defensible.

---

### 4. A quick experiment to probe the boundary

| Setup | Prompt length | Offer count | Lane | Metric |
|-------|---------------|-------------|------|--------|
| 1 | 10 k tokens (question + small offer set) | 5 | L2y | Accuracy / Oracle score |
| 2 | 10 k tokens (question + random filler) | 5 | L0 | Accuracy / Oracle score |
| 3 | 10 k tokens (question + large static doc) | 5 | L0 | Accuracy / Oracle score |
| 4 | 10 k tokens (question + small offer set) | 5 | L2y with prune‑to‑cold | Accuracy / Oracle score, token usage |

Run each across a held‑out question set where the next query is either *predictable* (same domain) or *unpredictable* (different domain). Measure whether the prune‑to‑cold lane still matches quality and how many tokens it saves.

If we see a clear “predictable‑recurrence” sweet spot, that would give us a concrete threshold for when we can justify governed eviction versus an infinite context strategy.
