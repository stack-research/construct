## Findings

**BLOCKING:** none.

**NON-BLOCKING:**

1. **Mapping-rule signature vs. sealed formula.** The sealed spec §2.5.5 writes the mapping as `expected_commitment_enum = mapping_rule(stratum, canonical_action_set)`, while the contract (`harness/efc_menu_composition_rules_v1.md` §2 and `harness/efc_menu_composition_v1.py` `derive_expected_enum`) implements `mapping_rule(stratum, canonical_action_set, role_map)`. Functionally the result is still total and deterministic: `role_map` is a fixture-level total assignment of every label to one of the frozen lexicon roles, and `check_fixture_composition` enforces `expected == derived` with refusal `expected_neq_mapping_output`. No per-fixture override door exists. However, the contract’s definition of `canonical_action_set` in §1 is strictly the ordered label list, so the role assignment is not part of the `(stratum, canonical_action_set)` input. For strict alignment with the sealed formula, a future revision could either fold the role assignment into the canonical action set or explicitly state that the hashed mapping rule consumes the fixture’s `role_map` as part of its input. This is notation/structure, not a mechanistic gap.

2. **Plausibility attestation exact-key check.** `harness/efc_menu_composition_v1.py` `_validate_plausibility_attestation` (lines 247–268) requires the attestation’s key set to equal exactly `{"fixture_id", "stratum", "reviewer_seat", "reviewed_at", "attestation_id"}` and rejects extra keys. The sealed spec §8.6 lists required keys and shape checks; this is stricter than required but not inconsistent. It correctly validates only presence and shape, never the plausibility content, as the contract honestly discloses in `harness/efc_menu_composition_rules_v1.md` §4.
