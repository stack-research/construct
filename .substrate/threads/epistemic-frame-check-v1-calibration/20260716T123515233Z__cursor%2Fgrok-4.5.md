### BLOCKING-1 — Full rendered prompt fingerposts stratum; larger leak surface than L2 audited; gap undisclosed / unpriced

I rendered and read all 15 prompts as the engine will receive them.

**Direct fingerpost:** every header is `[task {task_id} | population efc_calibration_v1]` and on this suite `task_id == fixture_id` with stratum-encoding prefixes:

| prefix | stratum | expected_commitment_enum |
| --- | --- | --- |
| `ir` | irrelevant | `maintain` |
| `mc` | match_commit | `proceed` |
| `mm` | match_mismatch | `suspend` |

Stratum name and expected enum do not appear as prose outside the menu, but the **task id alone** is a perfect stratum (hence expected-role) oracle. L2 never sees the header.

**Trigger-surface leak (also L2-invisible):** matching strata are uniformly `observation_boundary_present: False` + `cited_source`; irrelevant is `True` (or the allowed `internal_policy` variant on ir-05). That separates irrelevant vs matching for any engine that reads the declared trigger block. L2 inputs are only `task_body` + `action_set_ordering` (§8.6 / leak-audit contract). The full prompt also adds: header, full trigger surface, generic caution, commitment instruction.

**Conclusion:** the assembled prompt is a **strictly larger** leak surface than what the leak audit priced. Neither the D6a contract nor the build report discloses this under-approximation. Unpriced residual risk to `confounded(menu_leak)` honesty at pilot.

**Repair (any one, artifact-bound):** (i) opaque engine-visible task ids (no stratum codes) + re-attest if fixture bytes change; and/or (ii) drop/redact task_id from the scored foreground header; and/or (iii) extend the integrity leak gate to the engine-visible regions (header + trigger surface at minimum) with a dan/Sol ruling recorded in the contract. Until then D6a must not seal.

As shipped → **D6a incomplete / BLOCKING**. Minimum repair: add a contract section `menu_only` / deferred-mode MUST with a second hash slot and a hard refusal to pilot-contact without it; implementing the mode in this lifecycle is better.

(Using the current full render for `M_menu_only` would also confound §10.6.1: generic caution alone solicits the provenance tool.)

### BLOCKING-3 — Judgment call 1: scrub-then-check hole

`menu_labels_outside_action_set(..., fixture=fixture)` does `outside.replace(str(field), "")` for task_body + all trigger fields, then searches for menu labels.

**Hole (fresh RUN):** if a menu label equals an entire trigger-field value (e.g. `source_reference == label`), then a renderer injection of that label into generic caution is scrubbed away with the field value → **false pass**. Demo on a mutated fixture: injection visible without scrub; scrub reports `()`.

Live suite currently has no exact label==field equality (only substrings such as `release` inside `decision_scope` on ir-03/mc-02/mm-03; isolated injection of `release` into caution is still detected after scrub). The conformance vector is nevertheless unsound as a general injection detector.

**Ruling on call 1:** REJECT as stated. Repair: assert labels absent from **renderer-owned regions only** (frozen caution body, commitment instruction, section headers) — do not scrub fixture text globally.
