## Probe (1) — BUDGET-BREACH-BY-UNDERESTIMATE — **BLOCKING**

Pre-call estimate is `max(1, (utf8_len+3)//4)`. That is **not** conservative for code-/punct-dense or unicode-dense text (probe ratios ~1.3–2× underestimate vs plausible tokenization; adversarial transport can make actual ≫ estimate).

**Fresh RUN findings (MockTransport with inflated actual usage):**

1. **Mid-run overshoot then continue:** start near the I pool with estimate still under remaining headroom → first call pre-guard passes → actual usage pushes `input_tokens_spent` to **255000 > 250000** → runner records the overshoot as a normal completed row → only the *next* call gets `budget_refusal`. Ledger pins are hard ceilings; the crossing spend itself has no typed refusal.
2. **Accumulation:** from ledger start (I=18), actual=20k/call → after 13 calls `final_I=260018` (crossed) with prior calls all accepted; refusal only on the subsequent pre-guard.
3. **Last-call silent completion:** actual spike on call 30 → `status=completed`, `invocations=30`, `final_I=252900 > 250000`. **No `budget_refusal` at all.** The run reports green while the pinned input pool is breached in recorded actuals.

Root cause: `check_budget_guard` runs **only pre-call on estimates**; after `budget.input_tokens_spent += result.input_tokens` there is **no post-call actual ceiling check / halt**. Output and cost pools have the same structural gap (output at least uses max_output_tokens as a hard per-call projection; input does not).

Builder judgment 2 calling the heuristic "conservative" is **false** for stop-before-crossing. Disclosure of the estimate is fine; missing **post-call actual enforcement** is not.

**Required repair:** after every successful transport return, if any pool's *actual* spent exceeds its pinned ceiling → typed `budget_refusal` (or stricter: refuse the row as over-ceiling), STOP, and never report `completed` with overshot actuals. Pre-call estimate may remain a disclosed floor, but actuals are the ledger law.
