### Answer to your question 4 first, because it matters for what the oracle can carry

Fresh unchanged-scorer equality is **partially self-confirming, and the contract should say so**. The receipt already asserts `s1_digest == projected_s1_digest` and likewise for S2 — projection reproduces the source byte-for-byte, so scoring two byte-identical inputs with the same scorer must agree; the equality leg adds no discriminating power over the digest check on the happy path. Its real, non-circular value is elsewhere: it proves the `_unscored_pair` surgery (verdict stripping, `s1_ledger` rebinding) behaves identically on both sides, and it powers the sovereignty legs — the fork-identity tamper test shows the unchanged scorer, not the Core, catches what digests cannot. That is worth keeping, but the contract's success statement should attribute preservation to digest equality and reserve the scorer for sovereignty, so a future reader doesn't credit the oracle with a check it isn't performing.

### The blocking ground

**The M2 projection fails the exact probe class this same build repairs in X2, disclosed in my v0.1 review and claimed closed in your offer.** I appended a Core-legal, properly chained, un-receipted `placement_changed` (rogue controller, no `source_event_id`) flipping the `m2_earned_record` item hot→cold after ingest. `project_m2` succeeded; the carried state's placement now silently disagrees with what the adapter materialized, with no refusal anywhere. `project_m2` checks status, warrant health, one admission, one activation — and never placement.

In v0.1 I classified this shape as non-blocking residual coverage, because the defect class was newly discovered and no promise covered it. Neither condition holds now. The class is known; the repair standard exists in the sibling adapter in this very diff; the offer states the companion repair "closes the residual recorded in the v0.1 review" — while the second client, built after that repair, ships open to the identical probe. A known, closed defect class recurring in a simultaneously shipped sibling is a build-consistency defect, not a discovery. Endorsing it would establish that each new client may re-open the class until someone re-probes it, which inverts the fail-closed discipline the Core exists to carry.

Second finding, **disclosed-boundary note, not blocking**: a fabricated `metabolic_event` (`fabricated_help: 999`) on the m2 item also passes projection. `reported_metabolic_totals` is explicitly writer-reported accounting and nothing scores it, so this sits inside the declared boundary — but the M2 adapter never legitimately writes metabolic events at all, so the bounded repair below can close it for free.

### Proposed bounded repair (one, per budget)

In `project_m2`, after the existing admission/activation correspondence checks, refuse any lineage row that touches the `m2_earned_record` item outside the declared correspondence set: any `placement_changed` naming the m2 item (the M2 carry defines no legitimate placement change; require terminal placement `hot`), and any `metabolic_event` naming it (the adapter writes none). Ship one named refusal test per probe, mirroring the X2 guard's shape. No schema change, no new anatomy, no scorer contact; the ten-pair round-trip is unaffected. Optionally — and I'd support this as part of the same repair rather than new scope — hoist the shared rule into a small helper both adapters use: *events affecting adapter-materialized state must carry a valid source binding or be refused at projection*, so the third client cannot re-open the class by omission.
