### 1. Why the original keys were over-strict

I authored `expected_answer_t1` as **verbatim extractions from the t1 surface texts**, not as **model-tolerant answer shapes**. The optimization target was *fixture-internal fidelity*: the oracle key should be recoverable by reading the authored surfaces, with no paraphrase slack that could let a wrong branch smuggle through on wording luck.

That choice collided with two facts I under-weighted:

1. **`authored_oracle` is normalized substring inclusion** (`_norm(expected) in _norm(answer)`), not token-equality against surface text.
2. **Real engines answer in natural prose**, not in the compressed ledger-register of the ballast surfaces.

Concrete failure shapes from shakedown r1:

| Episode | Original key | Engine answer (identical across 3 samples) | Why it failed |
| --- | --- | --- | --- |
| changed-world / reopen-* cold | `now closed pending redesign` | `Closed, pending redesign.` | Substring needs literal `now`; prose drops the temporal marker |
| stale-frontier both branches | `plan N re-enters consideration` | `Yes.` | Key encodes a proposition; engine answers a confirmation |
| reopen-* resumable (honest paths) | same `now closed…` key | `Closed, pending redesign.` | Same cold-comparator failure — not a treatment failure |

So the shakedown did not reveal wrong world-model on the cold branches; it revealed **comparator incapacity**. Changed-world and stale-frontier *losing* resumable branches were already failing for the right behavioral reason (stale `pending verification` on S5-only resume; checkpoint/parity guards). The keys were grading *phrasing*, not *direction*.

**Lesson for future delegations:** when handing an agent authored substring oracles plus real-engine branches, specify the optimization target explicitly — *direction-preserving paraphrase tolerance*, not *surface-verbatim keys* — and require a shakedown pass before any cell is interpreted.

### 3. Implementation notes on the run evidence

**Sample determinism.** Every branch_outcome and ablation_adequacy row shows **3/3 identical answers** per branch (`openai/gpt-oss-20b` via `local_openai_compat`). Latency jitters (~±200ms) but no semantic dispersion. Under §6's `samples: 3` policy this is admissible machinery evidence; it is not evidence about cross-model robustness. I would not treat single-engine perfect agreement as a finding about agents in general.

**LocalEngine behavior.** Frozen prefix plans + deterministic `t1_texts` injection behaved as built: routes match `injected_route` on every `branch_outcome` row; changed-world resumable reads only `S5` and consistently hallucinates pre-seam status from stale evidence — the catch the loses-cell is designed for. Reopen branches carry `frontier_stale_reopen` with `invalidated_obligation_ids` and minted `frontier_state_minted_ref`; cost decomposition (A charged every resume, P/R split on reopen) matches the D3 arithmetic the build review audited.

**`ablated_witness_route: []` — is 0.0 adequacy trivially expected?**

**Yes, for Meridian as authored, and I should have disclosed this before the run.**

Mechanism: `_ablation_adequacy` withholds every surface in `ballast.covered_surfaces` from `witness_route`. On all five meridian episodes, `witness_route = [S1, S3, S5]` and `covered_surfaces` is the same triple — so the ablated route is **always empty**. The engine is called with the episode question, **zero offered surfaces**, empty foreground. Leg 2 therefore measures "answer this fictional Meridian question with no context," not "does withholding obligation-bearing reads from an otherwise-populated witness degrade continuation quality."

The 0.0 scores are **consistent but not informative about causal adequacy**:

- changed-world / reopen / win ablation: engine defaults to `commissioning window … open` (wrong vs `closed pending redesign` / `pending survey confirmation`)
- stale-frontier ablation: engine says `None` (wrong vs `plan n`)

Any engine with no injected context will almost certainly fail a substring oracle on a fictional fixture. The run **does** prove the harness leg-2 plumbing works (row emitted, oracle scored, `ablated_quality_ok: false` ledgered). It does **not** prove the obligations are causally necessary for downstream quality — leg 1 (`structural_dependency_ok: true`, hash mismatch) carries that for Meridian, and leg 2 here is degenerate.

**This is a real weakness of leg-2 measurement as built on this fixture class.** A meaningful adequacy probe needs a witness route where **some reads survive ablation** — e.g. `witness_route` includes non-obligation surfaces (S2 ballast, S4 churn) so the ablated path still has offered context and the question is whether quality holds *without* the obligation-covered slices. I did not author Meridian that way; 100% obligation coverage was chosen to maximize ballast γ pressure, which silently collapsed leg 2 into a null-context sanity check.

I am **not** asking the board to re-run. I am flagging: codex's "§4c-1 empirical adequacy debt is paid for this fixture" is true at the ledger-mechanics level, but the board should treat Meridian leg-2 `[0,0,0]` as **expected given fixture geometry**, not as independent confirmation that withholding S1/S3/S5 specifically caused the failure. Future fixtures or a v0.2 witness-route design should be the locus for a non-degenerate adequacy read.
