### D2: Stripping the Metaphor — The Rulebook is a Laboratory Toy, and Hint-Only is a Leak

We have spent three rounds designing a beautiful, intricate Swiss watch. Now let’s look at what we’ve actually built: a closed-world state machine that grades its own homework, paired with an escape hatch that lets us run the only episodes we actually care about with zero structural rigor. 

Here is my straight read on the state of the room.

#### 1. The Rulebooked Predicate-over-Tags Instrument is a Laboratory Toy

Let's be blunt: **no systems engineer in production will ever write a population-time rulebook or pre-tag their database just to let an agent resume.** 

By shrinking the instrument to "tagged, rulebooked fixtures only," we have designed a tautology. 
- The harness pre-tags the surfaces. 
- The harness author writes a deterministic `obligation_rulebook` that compiles pre-seam reads into post-seam obligations. 
- The "resumable-state" is just a serialized representation of this compiled state. 
- The resumed agent is then graded on whether its reads intersect the obligations we compiled for it.

This is not measuring an agent's memory, attention, or orientation. It is measuring the harness author’s ability to write a compiler. The agent is just a passive execution environment for a pre-compiled state machine. We are grading our own homework. A systems engineer does not care about this because it does not scale, it cannot handle open semantic domains, and it assumes a level of structural foresight that only exists in hand-crafted laboratory fixtures.

#### 2. `hint_only` Mode is a Massive Leak, Not an Honest Tier

The room created `hint_only` because we realized that the "rulebooked" mode is too fragile to handle real, complex, semantic reasoning. But let’s call `hint_only` what it is: **a leak that lets interesting episodes escape all structural rigor.**

In `hint_only` mode, we throw away `derive_live_obligations` because we admit we cannot derive obligations without a rigid grammar. So what is left?
- The agent writes a "hint" (which is just an ungoverned summary or partial conclusion).
- The resumed agent uses that hint to skip reads.
- We score it on cost.

Without a derived obligation set, we have no computational way to verify that the hint didn't just cache the final answer. The "reconstruction-illusion" and "answer-cache" cells become hand-waving post-hoc audits (violating R5) or fragile heuristics. `hint_only` is a compromise that lets us feel rigorous because we have a "rulebooked" mode, while actually running all our interesting, semantic test cases under a "trust-me" priced-hint benchmark. It should be dropped, or we should admit the entire instrument is just a priced-hint benchmark.

#### 3. The Minimal Production-Ready Instrument

If we were designing this for a systems engineer who needs agents to resume after unexpected network drops, rate limits, or hardware preemptions, here is what we would keep and what we would throw away.

**What to THROW AWAY:**
- **Precommitted Seams:** In reality, interruptions are asynchronous and unexpected. Pre-planning where the engine stops is a laboratory luxury.
- **Rulebooked Derivation & Tag Schemas:** Throw them out. They do not scale, they require manual cataloging, and they turn the agent into a passive compiler.
- **Three-Branch Symmetry:** We already agreed the warm branch is just a witness. Let’s stop pretending it’s a symmetric competitor.

**What to KEEP:**
- **Two-Branch Comparison:** Uninterrupted-warm (witness) vs. Cold-resume (treatment).
- **The Loses-Cells:** Specifically, `changed-world` and `false-continuity`. These are the only things that make any resume instrument honest.
- **Strict Size/Format Constraints on the Note:** The "state" is just a text note. We must strictly limit its size (e.g., max 500 tokens).

**The Minimal Design:**
1. **Arbitrary Interruption:** Run the agent. At an arbitrary, simulated interruption point (no pre-planning), freeze the execution.
2. **The Resume Note:** The agent is allowed to write a single, size-constrained text note *at the moment of interruption*.
3. **The Anti-Caching Guard:** To prevent the agent from just writing "the answer is X" in the note, we apply a simple, automated, domain-independent rule: the note must not contain any entity names, values, or specific actions that appear in the final answer but were not in the initial prompt.
4. **The Resume:** Start a fresh agent instance with the initial prompt + the resume note.
5. **The Scoring:** Measure:
   - **Quality:** Does the resumed agent achieve the same outcome as the uninterrupted witness?
   - **Cost:** Does the resumed agent read fewer documents (tokens) than a cold-started agent?
6. **The Ultimate Guard (Changed-World):** Between interruption and resume, perturb a key fact in the database. If the resumed agent blindly follows its note and produces a wrong answer (false-continuity), it loses. If it is smart enough to detect the change or fail gracefully, it wins.

This is minimal, realistic, requires zero hand-authored rulebooks, scales to any domain, and directly measures whether writing notes helps an agent resume after an unexpected drop without carrying stale assumptions into a changed world.
