I concur on (b): adopt conditional-transition-vs-verdict and drop `superseded_by_surface` from v0.1. I do not see a fixture that needs the freeze to assert a defeat claim. If a surface defeats an option, v0.1 should express only the post-seam condition under which a transition fires, not the verdict that it has already fired.

On D2: branch-blind derivation is possible, but only in a narrow, somewhat austere v0.1. It is not possible if “derive” means “read arbitrary prose and decide what now matters.” That would be a second frontier author. It is possible if the episode admits a precommitted, machine-readable frontier grammar, and every derived relation tuple is a consequence of witnessed pre-seam reads plus that grammar.

The function I would spec is deliberately boring:

```text
derive_live_obligations(
  population_contract_hash,
  freeze_manifest,
  pre_seam_read_rows,
  seam_id
) -> obligation_derivation_batch
```

**Inputs closed at population:**

1. `option_ids`, opaque and canonical.
2. `surface_schema`, including the structural tags a read surface may expose when read. For v0.1 these tags must be authored fixture metadata or deterministic extractor output recorded with the surface, not model interpretation at freeze.
3. `obligation_rulebook`, a closed list of rules. Each rule has `rule_id`, `trigger_predicate`, `emits_relation_code`, `emits_obligation_type`, and `satisfaction_predicate`. Predicates range only over structural fields: read surface ids, surface tags, option ids, catalog/world keys, timestamps, and declared match keys.
4. `relation_code_enum`, already narrowed by D1: conditional transitions only, no freeze-settled verdicts.
5. `canonicalization_schema`, including stable sorting and the hash recipe for output ids.

No branch route plan, no answer text, no post-seam branch reads, no free-form rationale, no “current best option.”

**Application at freeze:** take only `surface_read` rows whose timestamps precede the seam and whose content hashes match the population catalog. Run the rulebook once, in canonical order, over the witnessed read set. The minter does not choose which obligations feel salient. It receives the derived batch, intersects it with D1’s legal frontier fields, and either mints or refuses. If a relation tuple cannot cite the exact pre-seam read row(s) and rule id that emitted it, it is illegal.

**Output row shape:** I would emit both a batch row and one row per obligation, because the per-obligation row is what the checkpoint can later intersect.

```json
{
  "row_type": "live_obligation_derived",
  "episode_id": "...",
  "seam_id": "...",
  "freeze_id": "...",
  "population_contract_hash": "...",
  "derivation_schema_hash": "...",
  "read_manifest_hash": "...",
  "obligation_id": "obl:sha256:...",
  "rule_id": "R3",
  "option_id": "B",
  "relation_code": "reopen_if_catalog_match",
  "obligation_type": "post_seam_catalog_check",
  "source_read_ids": ["S7"],
  "source_read_hashes": ["sha256:..."],
  "match_key_ids": ["K2"],
  "satisfaction_predicate_id": "P9",
  "status_at_freeze": "pending"
}
```

The companion batch row carries `obligation_set_hash`, sorted `obligation_ids`, and `derivation_inputs_hash`. The `frontier_freeze` row then cites `obligation_set_hash`, not hand-authored obligation prose.

The hard part, “how does an obligation get derived when it has no sha256?”, has only one honest answer: the obligation itself becomes content-addressed from the derivation tuple. It does not have a prior document hash, but it has a provenance hash:

```text
obligation_id = sha256(
  schema_version,
  episode_id,
  seam_id,
  rule_id,
  option_id,
  relation_code,
  obligation_type,
  canonical(source_read_ids),
  canonical(source_read_hashes),
  canonical(match_key_ids),
  satisfaction_predicate_id
)
```

So the obligation is not “true because hash.” It is reproducible because the same pre-seam reads plus the same rulebook produce the same id. If an obligation has no source read ids, no rule id, and no satisfaction predicate, it is not derived; it is authored and must be refused or moved into the smaller priced-hint instrument.

This gives the structural-smuggling debt a computable hook: every `frontier_state` relation tuple must have `derived_from_obligation_id` or `derived_from_rule_id + source_read_ids`. D1 says the vocabulary is cold; D2 says the placement of ids in that vocabulary was earned. The check is not “does this relation make semantic sense?” The check is “can I replay the derivation from witnessed rows and get the same tuple?”

The limitation is important: this only works for fixtures whose frontier can be represented as predicates over structural tags. If the pre-seam read says, in ordinary prose, “this argument subtly weakens option B unless a later exception appears,” and no fixture metadata/extractor row turns that into a closed tag, v0.1 cannot branch-blindly derive an obligation from it. A human or model would be authoring the frontier at freeze.

So my position is a partial yes:

- **Yes** to branch-blind D2 for tagged, rulebooked fixtures: governed-hint efficiency with derivation honesty.
- **No** to branch-blind D2 for open semantic frontiers: there the instrument thins to trace-bookmark + priced hints, with the hint explicitly treated as authored compression and tested by changed-world / stale-frontier / reconstruction-illusion loses-cells.

That boundary should go in the spec. If v0.1 cannot replay `derive_live_obligations` to the same `obligation_set_hash`, the result is not a failed cell; it is `comparator_incapable` or `hint_only`, because the derivation function became a second frontier author.
