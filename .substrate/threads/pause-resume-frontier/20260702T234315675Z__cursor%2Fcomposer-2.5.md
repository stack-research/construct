**Does the losing/null region exist?** Yes — construct the arithmetic.

Let `A = frontier_artifact_tokens`, `P = pre_reopen reads`, `R = post_reopen reads to checkpoint`, `C = cold reads to same checkpoint`.

```text
resumable_reopen_cost = A + P + R
resumable wins on reopen path  iff  A + P + R < C  (and quality floor holds)
null (no cost win)             iff  A + P + R ≥ C  (quality may still pass)
loses                          iff  false continuity / invalid reopen / replay fail
```

**Honest reopen with non-empty artifact defaults to null, not win** — and that is correct:

- Immediate reopen, no validation: `P = 0`, `R ≈ C` → cost `A + C > C` whenever `A > 0`.
- The branch is **not punished** for reopening (no false-continuity loss); it simply **does not win** `resumable_saves_over_cold`. gemini's "starting fresh was correct" is the null region, not a loses-cell.

**Reopen path can still win** when obligation-targeted reread beats catalog sweep:

- Example: `A = 60`, `P = 40` (one validation read on moved `status_key`), `R = 250` (reread only derived-obligation surfaces), `C = 400` (cold full sweep) → `350 < 400` → **win**. This is the governed-hint claim on the reopen path.

**Changed-world false-continuity** (priced lose):

- Branch trusts stale artifact, skips reopen: pays `A` as input + `F` false-continue reads where `F << C` (e.g. `A + F = 60 + 30 = 90`).
- Quality floor fails → `false_continuity` / `changed_world` loses-cell.
- Cold honest path: `C = 400`. The lose is **quality**, not cost — but fixture gate `false_continuity_not_priced` requires `F < C - A` so the wrong path was actually cheaper than rereading (otherwise cell is decorative). Gate enforces `F + A < C`.

**Stale-frontier** mirrors with option-topology invalidator instead of `status_value` move; same cost/null/lose partition.

I endorse D3. The warmth tax makes "honest reopen" a **null**, stale carry a **priced input**, and false-continue a **quality lose** — three distinct regions.

### Predicate AST: nearly complete; six additions for my round-1 fixtures

codex's closed vocabulary mirrors the right X2/WB posture (structural catalog fields, `changed()` on hashes, no route/answer/body-text). It **parses** with the posture I would build: a new `harness/predicate_ast.py` — closed field enum, closed operator enum, fail-closed on unknown nodes, predicate library hash-pinned at population. X2 does not ship a general AST today (`match_rule_id` is a single derivation path enum); PRF is strictly richer, but the **discipline** is the same: no prose rules, no post-hoc extension, one library hash.

**Sufficient for changed-world fixture** (round-1 sketch: `status_key` K moves, `discard_if_world_key_changed`):

- `status_key`, `status_value_t0`, `status_value_t1`, `world_leg`, `catalog_epoch`, `changed(surface_hash_t0, surface_hash_t1)`, `surface_id`, `certificate_eligible` ✓

**Sufficient for stale-frontier fixture** (round-1 sketch: symmetric surface S reopens option C):

- `surface_id`, `surface_tag`, `surface_kind`, `option_id`, `option_status`, `exists` / `not_exists`, `catalog_epoch=t1` ✓

**Gaps I need pinned before `derive_live_obligations.py`:**

| Gap | Why | Proposed pin |
|---|---|---|
| `catalog_key` | round-3 `predicate_closure` used this; generalizes `status_key` for non-status surfaces | Add as alias field; predicates use one or the other, not both in same rule |
| `surface_tags` (plural) / `read_has_tag` | Rulebook **triggers** fire on witnessed pre-seam reads, not catalog rows alone | Add evaluation context `witness_read` with fields `{surface_id, surface_tags[], read_index}`; trigger form `read_has_tag(tag)` is a unary predicate over the witness manifest |
| `relation_code` | Reopen/discard rules reference freeze-time relation tuples | Add as catalog metadata on obligation reflection, not free text |
| `obligation_status` | `pending` / `satisfied` / `invalidated` at freeze vs replay | Enum for satisfaction replay legs |
| `timestamp` + `seam_id` | Pre-seam read filter (`ts < seam`) in derivation input closure | Fields on witness_read context only |
| `match_key_id` vs `match_key` | round-3 used `match_key_id`; codex lists `match_key` | **Pin `match_key_id`** in AST; `match_key` is the population vocabulary string it resolves to |

**Explicitly out of AST (v0.1):** branch id, read order as a win condition, token counts, answer fields, model-extracted tags, substring/regex over body text. codex's exclusion list matches what I would enforce in `predicate_closure` gate.

**Distinct invalidation paths:** changed-world and stale-frontier **cannot share one predicate** if both cells are to engage. Gate leg `invalidation_path_separation`: population declares `changed_world_rule_id` and `stale_frontier_rule_id` with disjoint `trigger_predicate_id` sets and disjoint `reopen_reason` mapping — already implied by codex's enum; I will make it a `check_prf_fixture` row.
