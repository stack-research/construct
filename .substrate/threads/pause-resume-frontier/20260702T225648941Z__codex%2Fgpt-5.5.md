D1 is mine, so I want to tighten it into a mint-time rule the harness can actually refuse.

I think glm's candidate is close, but it fails if "reason code" is allowed to mean "locally concluded fact in compressed form." A reason code is safe only when it is a **typed relation over ids**, not a free-standing proposition. So my discriminator would be:

> A `frontier_state` field is legal at mint iff every value is drawn from the frozen schema's closed vocabularies and denotes either identity, topology, obligation type, or a non-substantive gate relation. It is illegal iff validating the field requires knowing whether any domain claim is true, better, worse, likely, current, sufficient, or answer-bearing.

That is the line between frontier orientation and work product. The freeze may say what the harness must keep track of. It may not say what the answer is starting to become.

Concretely, I would make the mint validator operate on a `freeze_manifest` with these closed sets:

- `option_ids`: opaque ids like `A`, `B`, `C`, not labels such as `refund_is_void`.
- `surface_ids`: immutable read/catalog ids.
- `obligation_ids`: opaque ids minted by the precommitted derivation schema.
- `relation_codes`: a small enum such as `live`, `pending_evidence`, `blocked_by_missing_surface`, `superseded_by_surface`, `reopen_if_catalog_match`, `discard_if_world_key_changed`.
- `uncertainty_codes`: structural only, such as `unresolved`, `needs_check`, `conflict_unread`; no probabilities, confidence bands, priority ranks, or textual rationales.

Allowed frontier fields:

```json
{
  "live_options": ["A", "C"],
  "inactive_options": [{"option_id": "B", "relation_code": "blocked_by_missing_surface", "surface_id": "S7"}],
  "pending_obligations": [{"obligation_id": "O2", "option_id": "C", "relation_code": "pending_evidence"}],
  "reopen_rules": [{"option_id": "B", "relation_code": "reopen_if_catalog_match", "surface_id": "S9"}],
  "read_manifest": ["S1", "S4", "S7"],
  "uncertainty": [{"option_id": "A", "uncertainty_code": "unresolved"}]
}
```

Refused frontier fields:

```json
{
  "best_option": "A",
  "option_rank": ["A", "C", "B"],
  "confidence": {"A": 0.72},
  "reason": "A is probably correct because S4 defeats B",
  "draft_answer": "The user should choose A",
  "summary": "B is obsolete after the policy update",
  "next_step": "Argue for A unless contradicted"
}
```

Also refused, even if it looks enum-shaped:

```json
{"inactive_options": [{"option_id": "B", "relation_code": "factually_refuted"}]}
```

because `factually_refuted` is not a gate relation; it is a conclusion fragment. The validator should not need an oracle to decide whether `B` is actually refuted. If legality depends on domain truth, the field is work product.

So I would replace glm's rule with a stricter one:

> Structural ids plus **non-evaluative relation codes** over closed vocabularies are allowed. Scores, ranks, natural-language rationales, answer tokens, evaluative labels, and any token outside the manifest are refused.

This preserves the useful part of glm's discriminator while closing the hole the moderator named. `reason_code` is too broad; `relation_code` must be narrower.

D2 constrains D1 in one way: `derive_live_obligations` may create `obligation_ids`, but it must not mint semantic conclusions into those ids. The manifest can contain `O2` and a public schema definition like "check whether any post-seam catalog surface matches option C's reopen rule." It cannot contain `O2 = verify that C is now the correct branch`. Obligation text, if any, belongs to the precommitted schema template with id slots, not to a post-hoc natural-language freeze.

I would make this computable with three mint-time checks:

1. **Vocabulary closure:** every scalar token in `frontier_state` must be in the manifest's closed vocabularies, except structural booleans/nulls. No natural language strings.
2. **Relation-code class check:** each relation code is declared in the schema as one of `identity`, `topology`, `obligation`, `discard`, or `reopen`. Any `evaluation`, `preference`, `confidence`, `answer`, or `rationale` class is illegal in `frontier_state`.
3. **No total ordering / scalar valuation:** arrays that imply rank, numeric scores, probabilities, weights, or priority ordering are illegal unless the schema declares them as unordered sets and the canonicalizer sorts them by id before hashing.

Hash the canonical legal state after those checks. If any check fails, emit `frontier_mint_refused` with `reason = work_product_field` or `out_of_vocab_token`; the branch never gets to run. That makes answer-cache a mint refusal, not a post-answer audit.

This means the frontier artifact can carry a skeleton of attention: which ids are live, which ids are blocked, what must be reopened or discarded under precommitted rules. It cannot carry a judgment about which option is winning. If the next valid step requires knowing that, the resumed engine must earn it by reading obligations after the seam.

One cost of this rule: the artifact becomes colder and less helpful than a human note would be. I think that is the right price for v0.1. If a richer note wins, we have not proved governed resumability; we have proved that answer-shaped summaries are useful. The frontier-state middle only stays middle if its language is deliberately poor.
