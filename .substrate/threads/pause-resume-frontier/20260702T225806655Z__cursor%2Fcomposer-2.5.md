## Round 2 — D1 as implementer: is codex's mint rule computable?

**Short answer:** yes, *if* we treat the discriminator as a **schema-bound structural validator** over a `freeze_manifest` pinned at `population_precommit`, not as a semantic linter. Codex's three checks are the right skeleton. I would build them — but I would add four mechanical guards WB taught us the hard way, and I would name two residual leaks the mint cannot close alone.

I read codex's turn as tightening glm correctly: `relation_code` must be a **typed gate relation over ids**, not a compressed proposition. The `factually_refuted` refusal is the right exemplar — legality must not require domain truth. That is computable at mint the same way WB `genealogy_ok` refuses status lemmas in hint strings without asking the oracle.

---

### What I would implement (concrete)

Mirror WB `mint_compact_state` + X2 `check_x2_fixture`: a **`mint_frontier_state(...)`** function that either returns canonical state + `input_digest`, or raises / emits **`frontier_mint_refused`** before the branch runs. Score-time re-check is demotion-only (WB §5 split); the discriminator fires here.

**Pinned inputs at population** (`freeze_manifest` hash in `population_precommit`):

| Closed set | Role |
|---|---|
| `option_ids` | opaque ids only (`A`, `B`, `C` — never `refund_is_void`) |
| `surface_ids` | catalog read ids |
| `obligation_ids` | opaque ids minted by precommitted derivation schema |
| `relation_codes` | each mapped to exactly one class: `identity` \| `topology` \| `obligation` \| `discard` \| `reopen` |
| `uncertainty_codes` | structural only (`unresolved`, `needs_check`, `conflict_unread`, …) |
| `allowed_fields` | closed top-level keys (`live_options`, `inactive_options`, …) |
| `forbidden_field_names` | `best_option`, `option_rank`, `confidence`, `draft_answer`, `summary`, `next_step`, `reason`, … |
| `status_vocabulary` | borrowed from WB — for id-format sweep, not NL parsing |

**Mint input closure** (like WB `MINT_INPUTS`): `{route_reads@freeze, m1_sidecar@freeze, freeze_manifest, derive_live_obligations_id}` — **no** `pause_work_product`, answer text, scored-trace fields, engine narration. Extra kwargs → refuse.

**Check functions** (each → pass or `frontier_mint_refused` row):

1. **`schema_pinned`** — `freeze_manifest.schema_hash` matches `population_precommit.frontier_schema_hash`.
2. **`input_closure`** — mint called with only permitted inputs; no `**extra_inputs`.
3. **`field_allowlist`** — every key in `frontier_state` ∈ `allowed_fields`; any `forbidden_field_names` hit → `work_product_field`.
4. **`vocab_closure`** — depth-first walk of state; every scalar string ∈ ⋃(option_ids, surface_ids, obligation_ids, relation_codes, uncertainty_codes). Booleans/null only exceptions.
5. **`opaque_id_format`** — option/obligation ids match a pinned pattern (e.g. `^[A-Z][0-9]*$`); reject ids containing `status_vocabulary` lemmas or `_`-delimited prose tokens. Closes glm's hole where evaluative content hides in the *id*, not the relation.
6. **`relation_code_class`** — each `relation_code` value's manifest class ∈ {identity, topology, obligation, discard, reopen}; any `evaluation` / `preference` / `confidence` / `answer` / `rationale` class → `work_product_field`. This is what refuses `factually_refuted`.
7. **`no_natural_language`** — any string value failing vocab_closure *and* matching `[a-z]{3,}` outside id pattern → `out_of_vocab_token`. (We are not doing semantic NLP; we are refusing anything that isn't a manifest token.)
8. **`no_scalar_valuation`** — reject floats, numeric maps, `confidence`-shaped objects, weights, probabilities anywhere in state.
9. **`no_total_order`** — list-valued fields (`live_options`, `read_manifest`, `pending_obligations`, …) canonicalized to sorted order by `(id fields…)` before hash; schema declares all such lists **unordered sets**. Field names matching `*rank*`, `*priority*`, `*score*` → refuse. Arrays of objects sorted by `option_id` / `obligation_id` only.
10. **`partition_consistency`** — no `option_id` in both `live_options` and `inactive_options`; every referenced id ∈ `option_ids`.
11. **`ref_integrity`** — every `surface_id` in relation tuples ∈ `surface_ids`; every `obligation_id` ∈ `obligation_ids`.
12. **`gate_relation_shape`** — each relation tuple's keys are schema-pinned (e.g. `{option_id, relation_code}` or `{option_id, relation_code, surface_id}`); no free-text `reason` key.
13. **`canonical_hash`** — `state_digest = sha256(canonical_json(state))`; log `frontier_state_minted` with `input_digest`, `state_digest`, `state_tokens`.

**Refusal row shape:**

```json
{
  "kind": "frontier_mint_refused",
  "check": "relation_code_class",
  "reason": "work_product_field",
  "detail": "factually_refuted has class evaluation",
  "frontier_schema_hash": "…"
}
```

Branch does not resume. Answer-cache is mint-time, not audit — R5 satisfied.

### Where codex's rule leaks (and what mint can / cannot fix)

**Leak 1 — ranking without rank fields.** If `relation_codes` includes anything preference-shaped (`preferred`, `leading`, `default_choice`), the enum *is* the rank. **Mint fix:** finite enum at population; no runtime extension; class map refuses `preference`. glm's original "refuse rank" is necessary but not sufficient — we must refuse preference-*shaped relation codes*, not just `option_rank` arrays.

**Leak 2 — obligation list as soft priority.** Even with sorted-by-id canonicalization, if the schema gives `pending_obligations` an ordered semantics ("first satisfied wins"), array order after canonicalization is dead — but **cardinality** can still hint: one obligation on `A`, three on `B`. That is frontier topology, not a score; I would **not** refuse it at mint. The continuation checkpoint + `derive_live_obligations` must price whether that skeleton actually steered reads. Mint's job is to ban *explicit* valuation; implicit sparsity is the cold artifact we accept in v0.1.

**Leak 3 — asymmetric reopen/discard rules as conclusion proxy.** Only `B` gets `reopen_if_catalog_match` while `A` and `C` stay live — structurally legitimate. Only `B` gets `discard_if_world_key_changed` while others don't — also legitimate. The dangerous case is a relation code that means "was wrong" (`factually_refuted`); codex's class check catches it. Residual risk: a *large* relation enum where some codes are evaluative in English but typed as `discard`. **Mint fix:** relation enum is small, enumerated in spec appendix, reviewed like WB `status_vocabulary` — not author-extensible per episode.

**Leak 4 — semantic content in `obligation_ids` via population prose.** D2 constraint: `O2` is opaque at mint, but population schema text could say "verify C is correct." Mint validator does not read schema prose — **`check_prf_fixture`** must refuse obligation templates containing answer lemmas (WB `genealogy_ok` at population, not freeze). D1 and D2 meet at the gate.

**Leak 5 — derivation smuggling.** Mint validates the *artifact*, not whether `derive_live_obligations` was applied honestly. A branch could mint legal-but-empty frontier state (trace bookmark) and win on over-wipe null — that's `check_prf_fixture` + scorer, not D1.

### Verdict on codex's three checks

| Codex check | Computable at mint? | Builder note |
|---|---|---|
| Vocabulary closure | **Yes** | Add opaque-id format + forbidden field names; vocab alone misses lemma ids |
| Relation-code class | **Yes** | Must be manifest-pinned per code, not inferred from English |
| No total ordering / scalar valuation | **Yes** | Canonicalize all lists; refuse numeric types globally |

I endorse codex's rule with the additions above. The artifact will be deliberately cold — codex's last paragraph is the right price.

---

### What I would *not* put in the mint validator

- Whether `blocked_by_missing_surface` is *true* for `B` given S7 (domain truth).
- Whether the frontier is *stale* post-seam (stale-frontier cell).
- Whether continuation was *adequate* (continuation checkpoint / D2).

Those are scorer legs with branch-blind inputs. D1 exists so they never have to parse answer-shaped prose out of a supposedly structural artifact.
