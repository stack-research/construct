## Earliest failed boundary

**§8.5 target-stratum definitions silently conflate the structural trigger with the syntactic inputs that the `A_always_check` lane must receive, forcing the irrelevant stratum to be either untriggerable or check-runnable, not both as the spec requires.**

Cite:
- §2.1 predicate: `source_reference_present == true AND decision_scope_present == true` are two of the four trigger conjunctions.
- §8.2 lane board: `A_always_check` — "Same named check runs on every task; resident selectivity unused."
- §8.5: "Every task, including the irrelevant stratum, supplies syntactically valid inputs to the named check so `A_always_check` executes the same action rather than a cheaper substitute."
- §8.5: "Irrelevant" stratum has `Trigger: does not fire` and `Check result: not run` (for the candidate lane), but `A_always_check` must still run it.

Contradiction:
1. The check `scope_provenance_check_v0` (§2.2) needs a `source_reference` and a `decision_scope` to compare against the fetched provenance record.
2. If the irrelevant task has both fields present, the §2.1 predicate fires (because presence is a conjunction member), contradicting the irrelevant stratum's `does not fire` requirement.
3. If the irrelevant task lacks either field, the check cannot run with syntactically valid inputs, contradicting the §8.5 requirement for `A_always_check`.

A task cannot simultaneously satisfy both requirements unless the predicate depends on something other than the check's own syntactic inputs. The spec tries to escape this by saying irrelevant tasks are "surface-similar to matching tasks" while not triggering, but the only structural surface the predicate inspects is the very presence of the inputs the check needs. There is no second-order surface feature named that can make a task check-runnable without being trigger-runnable.

## Impact

This is not a Part II implementation detail. It is a Part I boundary definition that makes the lane board and stratum definitions jointly unrealizable:
- The `irrelevant` silence loses-cell (§9.2.4) and the always-check comparator (§8.2, §9.2.5, §9.4) both depend on a population of tasks where the candidate controller is silent and the always-check lane is not a cheaper substitute.
- If the fixture author satisfies the predicate by removing `source_reference` or `decision_scope` from irrelevant tasks, `A_always_check` cannot execute the named check and becomes a no-op substitute, violating the comparator design.
- If the fixture author keeps the fields so `A_always_check` runs, the candidate controller also fires on those tasks, violating the zero-false-fire gate in §9.1 and making the silence loses-cell impossible.
- The family-validity gate "false fires and missed fires are exactly zero at the deterministic controller layer" (§9.1) cannot be met for a non-empty irrelevant stratum under the current predicate.

## Minimal repair

Add a structural discriminator to the predicate that is **not** required by the check itself, so the irrelevant stratum can carry the check's syntactic inputs without triggering the candidate controller. Concretely, amend §2.1 and §8.5 together:

```text
predicate = assertion_basis_kind == cited_source
            AND observation_boundary_present == false
            AND source_reference_present == true
            AND decision_scope_present == true
            AND trigger_guard_present == true
```

where `trigger_guard_present` is a task-surface flag that is part of the matching-task signature but is not consumed by `scope_provenance_check_v0`. The irrelevant stratum must then be defined as: same `source_reference` and `decision_scope` shape, but `trigger_guard_present == false` (or `assertion_basis_kind != cited_source` with a fallback syntactic wrapper that still populates the check inputs). The exact discriminator is a fixture-design choice, but Part I must require one and assert that the check inputs are orthogonal to it.

Alternatively, and more minimally: change §8.5 so that `A_always_check` receives the check inputs only when the candidate trigger fires, while an `A_forced_check` lane receives syntactically valid inputs on every task by construction. Then the always-check cost comparator is cleanly separated from the candidate's silence claim. But the simplest repair is to name a trigger-only structural guard in §2.1.

## Secondary, verified block: §10.3 MIE / detectability pins are arithmetically incompatible with the non-inferiority and efficiency-win thresholds they are supposed to make measurable
